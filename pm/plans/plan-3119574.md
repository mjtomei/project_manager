# Plan: Walker UI for the Augmented Adversarial-Review Cycle

The augmented cycle (`pm/docs/adversarial-review/METHODOLOGY.md` § The augmented cycle) produces three artifacts per cycle: `REVIEW_CYCLE_N.md`, `CITATION_AUDIT_CYCLE_N.md`, `REVIEW_RESPONSE_CYCLE_N.md`. The response file carries proposed changes — some flowing from review findings (`provenance: reviewer-comment`), some from per-citation audit entries (`provenance: audit-entry`). The walker is the human acceptance surface between the response session and the apply step.

The walker is **optional** — auto-run mode applies the response session's recommendations directly. The walker exists for when the human wants to read, accept, edit, or reject proposed changes before they land.

The cycle itself runs in a Claude session via its normal tool use; the Python side is the walker UI, the CLI commands to launch sessions, and the markdown format primitives.

## Scope

- **CLI: `pm review <target>`** — launches a Claude session in a new tmux pane with augmented-cycle methodology context against a target file/topic.
- **CLI: `pm plan literature-review <plan>`** — launches a literature-review session as a pane inside an existing plan's tmux window. TUI plans-pane keybinding wires up the same command.
- **CLI: `pm review ui [--port]`** — launches the walker server.
- **Walker UI** — proposed-changes walker + review-comments browse + citation-audit browse + citations status view + dashboard + general-comments side panel. The current cycle's review comments and audit entries are viewable read-only and live-update as they are written, before `awaiting-human-review`; a running stage shows an animated activity indicator.
- **Markdown format primitives** — response-block parser/writer, interaction-log appender, audit-doc parser, response-doc parser with provenance tags, state-file parser/writer (`STATE.md`), focus-file parser/writer (`UI_FOCUS.md`), notes-file appender (`NOTES.md`).
- **Project registry** — reviews become first-class entries in `pm/project.yaml` alongside plans and PRs, with a per-review directory under `pm/docs/adversarial-review/reviews/<id>/`.
- **QA enablement** — a QA instruction file (fixture review setup + phase-driving), a web-UI recording artifact recipe (full browser video via headless Chrome + HTTP/SSE protocol capture), and a container PR pre-installing Playwright + Chromium + ffmpeg, so the browser-heavy walker PRs become runnable under pm's Claude-driven QA flow instead of human-only.

## Goals

- Make the response→apply gap a UX-rich surface so humans can comfortably accept/edit/reject without leaving canonical markdown as the source of truth.
- Keep the cycle runnable headless (auto-run mode) so the walker is value-add, not required.
- Real-time bidirectional sync between session and walker via filesystem watches + SSE (sub-100ms reactivity).
- Treat reviews as registered project entities so they can be enumerated, resumed, and cross-referenced from plans.

## Key design decisions

- **Code under `pm_core/`; `pm/` is project data only.** No Python lives under `pm/` — it holds `project.yaml`, plans, and the per-review directories under `pm/docs/adversarial-review/reviews/`.
- **Decision storage: inline response-block on each proposed change.** The response file is the canonical decision record. Alternative considered: sibling `*.decisions.md` file — rejected because it forces two-file reads and detaches decisions from source.
- **Bulk-accept default scope: current filter.** Whole-document bulk-accept is opt-in via an explicit "expand scope" toggle — avoids the "accidentally accepted 80 things" footgun.
- **Cycles are independent.** Each cycle's review, audit, and response run as fresh blind sessions over the current cycle's artifacts only — no cross-cycle conflict detection or prior-cycle commitment tracking. There is no mechanism for one cycle to reconcile against another; independence is the design.
- **Read-canonical, write-structured.** Walker writes back into the response file's response blocks (one fenced HTML comment per proposed change). No sidecar decision files.
- **In-progress cycles are viewable, not editable.** The walker surfaces the current cycle's review comments and audit entries live (streamed via SSE) during the `review` / `audit` / `response` phases, so the human can read findings before the response and Apply gate exist. Viewing is always allowed; the `awaiting-human-review`-only restriction governs *modification*, not visibility. A running stage shows an animated activity indicator (the audit phase animates a live "K citations audited" count).
- **Single UI → session signal: the Apply button.** Clicking Apply writes `STATE.md` transitioning `awaiting-human-review` → `applying`. Session watches the same file and proceeds. That's the entire UI → session communication channel.
- **Multi-UI state ownership: single writer via leader lock.** Multiple `pm review ui` processes may render the same review (reads are lock-free, torn-read-safe via atomic-rename writes), but only one may write `STATE.md` — the Apply transition is a check-then-write that two UIs could otherwise both pass and double-fire. Each process holds an `flock(LOCK_EX|LOCK_NB)` on a `STATE.md.leader` lockfile; the holder is the writer. Failover is automatic: the OS releases the flock on process death and a follower's periodic retry takes the mantle, with an SSE `leader` event flipping Apply on/off on the right clients. This leader lock (who *may* write state) composes with — and is distinct from — `md_writer`'s per-write lock (which serializes the write itself). Edit controls are not leader-gated; only Apply is. Implemented in pr-d887f4c (see its PR note + spec R10).
- **Local-only.** FastAPI single-file server, vanilla JS + one CSS file, no build step. `localhost:<port>`.
- **State is purely file-backed.** Markdown is the database. `STATE.md` carries cycle/phase/mode; `UI_FOCUS.md` carries session-driven view-direction; `NOTES.md` is the free-text journal.
- **Per-review directory layout.** Each registered review gets `pm/docs/adversarial-review/reviews/<id>/` containing `STATE.md`, `UI_FOCUS.md`, `NOTES.md`, and per-cycle files. Pre-existing top-level audit/cycle files stay as historical archives — no backwards compatibility for the audit-browse view.

## Constraints

- No Python iteration runner; the Claude session runs the cycle via its normal tool use.
- No persistent database; markdown is canonical.
- Single user, local-only — no multi-user collaboration.
- Sub-100ms walker reactivity to filesystem changes (via `watchdog` + SSE).
- `pm/project.yaml` writes must round-trip-preserve other top-level keys (`plans`, `prs`).
- Walker modifications allowed **only** in the `awaiting-human-review` phase of the current cycle. Every other state renders read-only badges.

## Project registration and directory layout

`pm/project.yaml` gains a `reviews:` list:

```yaml
reviews:
  - id: regression
    target: pm/plans/plan-regression.md
    target-type: plan       # plan | file | topic
    status: active          # active | archived
```

Per-review directory:

```
pm/docs/adversarial-review/
  METHODOLOGY.md
  CITATION_USE_AUDIT.md
  CITATION_CRAWL.md
  reviews/
    <review-id>/
      STATE.md
      UI_FOCUS.md
      NOTES.md
      REVIEW_CYCLE_1.md
      CITATION_AUDIT_CYCLE_1.md
      REVIEW_RESPONSE_CYCLE_1.md
      ...
```

Resume-vs-create: registry hit + `active` → resume; hit + `archived` → warn; miss → create entry + directory + initial `STATE.md`.

## Markdown formats

### Response-block (per proposed change in `REVIEW_RESPONSE_CYCLE_N.md`)

```markdown
<!-- proposed-change
id: change-<n>
provenance: reviewer-comment   # reviewer-comment | audit-entry
source-anchor: <link>
target-section: §3 — sycophancy framing
before: |
  <verbatim current passage>
after: |
  <proposed replacement>
suggested-verdict: accept       # accept | reject | modify
suggested-rationale: |
  <reasoning>
human-verdict:
human-rationale:
human-commentary:
status: pending                 # pending | accepted-as-suggested | edited | skipped | auto-accepted
interactions:
-->
```

### `STATE.md`

```yaml
current-cycle: 3
current-phase: awaiting-human-review
# review | audit | response | awaiting-human-review | applying | complete
mode: human-reviewed              # auto-run | human-reviewed
last-transition: 2026-05-20T14:32:00Z
```

### `UI_FOCUS.md`

```yaml
view: audit-browse            # changes | audit-browse | citations | dashboard | notes
cycle: 3
target: citation-2024-xxxxx
timestamp: 2026-05-20T15:30:00Z
```

### `NOTES.md`

Free-text journal, section-tagged, append-only with timestamps. Loadable into session context so notes influence the next cycle's response.

## Phases & lock states

| Phase | Breadcrumb (current cycle) | Human can |
|---|---|---|
| `review` | `Cycle N · review in progress` (animated) | View current-cycle review comments as they stream (read-only) or browse prior cycles |
| `audit` | `Cycle N · audit loop running · K citations audited` (animated) | View current-cycle review + partial audit entries as they stream (read-only) or browse prior cycles |
| `response` | `Cycle N · response in progress` (animated) | View current-cycle review + audit (read-only) or browse prior cycles |
| `awaiting-human-review` | `Cycle N · ready for your review · editable` | Walk changes, click **Apply** |
| `applying` | `Cycle N · applying accepted changes` (animated) | View current-cycle artifacts (read-only) or browse prior cycles |
| `complete` | `Cycle N · complete · read-only` | Browse history |

Prior cycles are always read-only. The current cycle's in-progress artifacts (review, audit) are also read-only until `awaiting-human-review` — they are viewable and live-updating but not editable. A running stage (`review` / `audit` / `response` / `applying`) shows an animated activity indicator in the breadcrumb and Status panel.

## Server-pushed updates

Walker server uses `watchdog` to watch per-review `STATE.md`, `UI_FOCUS.md`, and the current cycle's `REVIEW_CYCLE_N.md`, `CITATION_AUDIT_CYCLE_N.md`, and `REVIEW_RESPONSE_CYCLE_N.md`. Watching the review and audit files (not just the response) is what lets the review-comments and audit-browse views update live during the `review` / `audit` / `response` phases. Filesystem events trigger SSE pushes on `/events?review=<id>`. No polling. Session watches `STATE.md` symmetrically for the Apply transition.

## Non-goals

- Python iteration runner.
- Auto-run as a CLI flag on a Python command (auto-run is the human telling the session to loop).
- Multi-user collaboration.
- Persistent database.
- Backwards compatibility with pre-flow `CITATION_AUDIT_*.md` files.
