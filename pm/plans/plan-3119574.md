# Plan: Walker UI for the Augmented Adversarial-Review Cycle

The augmented cycle (`pm/docs/adversarial-review/METHODOLOGY.md` Â§ The augmented cycle) produces three artifacts per cycle: `REVIEW_CYCLE_N.md`, `CITATION_AUDIT_CYCLE_N.md`, `REVIEW_RESPONSE_CYCLE_N.md`. The response file carries proposed changes â€” some flowing from review findings (`provenance: reviewer-comment`), some from per-citation audit entries (`provenance: audit-entry`). The walker is the human acceptance surface between the response session and the apply step.

The walker is **optional** â€” auto-run mode applies the response session's recommendations directly. The walker exists for when the human wants to read, accept, edit, or reject proposed changes before they land.

The cycle itself runs in a Claude session via its normal tool use; the Python side is the walker UI, the CLI commands to launch sessions, and the markdown format primitives.

## Scope

- **CLI: `pm review <target>`** â€” launches a Claude session in a new tmux pane with augmented-cycle methodology context against a target file/topic.
- **CLI: `pm plan literature-review <plan>`** â€” launches a literature-review session as a pane inside an existing plan's tmux window. TUI plans-pane keybinding wires up the same command.
- **CLI: `pm review ui [--port]`** â€” launches the walker server.
- **Walker UI** â€” proposed-changes walker + citation-audit browse + citations status view + dashboard + general-comments side panel.
- **Markdown format primitives** â€” response-block parser/writer, interaction-log appender, audit-doc parser, response-doc parser with provenance tags, state-file parser/writer (`STATE.md`), focus-file parser/writer (`UI_FOCUS.md`), notes-file appender (`NOTES.md`).
- **Project registry** â€” reviews become first-class entries in `pm/project.yaml` alongside plans and PRs, with a per-review directory under `pm/docs/adversarial-review/reviews/<id>/`.

## Goals

- Make the responseâ†’apply gap a UX-rich surface so humans can comfortably accept/edit/reject without leaving canonical markdown as the source of truth.
- Keep the cycle runnable headless (auto-run mode) so the walker is value-add, not required.
- Real-time bidirectional sync between session and walker via filesystem watches + SSE (sub-100ms reactivity).
- Treat reviews as registered project entities so they can be enumerated, resumed, and cross-referenced from plans.

## Key design decisions

- **Code under `pm_core/`; `pm/` is project data only.** No Python lives under `pm/` â€” it holds `project.yaml`, plans, and the per-review directories under `pm/docs/adversarial-review/reviews/`.
- **Decision storage: inline response-block on each proposed change.** The response file is the canonical decision record. Alternative considered: sibling `*.decisions.md` file â€” rejected because it forces two-file reads and detaches decisions from source.
- **Bulk-accept default scope: current filter.** Whole-document bulk-accept is opt-in via an explicit "expand scope" toggle â€” avoids the "accidentally accepted 80 things" footgun.
- **Auto-run conflict handling.** Response session is responsible for noticing prior-cycle commitments. If a conflict slips through, in auto-run the change is left `pending` with a `low-confidence-suggester` flag; in walker mode the human picks.
- **Read-canonical, write-structured.** Walker writes back into the response file's response blocks (one fenced HTML comment per proposed change). No sidecar decision files.
- **Single UI â†’ session signal: the Apply button.** Clicking Apply writes `STATE.md` transitioning `awaiting-human-review` â†’ `applying`. Session watches the same file and proceeds. That's the entire UI â†’ session communication channel.
- **Local-only.** FastAPI single-file server, vanilla JS + one CSS file, no build step. `localhost:<port>`.
- **State is purely file-backed.** Markdown is the database. `STATE.md` carries cycle/phase/mode; `UI_FOCUS.md` carries session-driven view-direction; `NOTES.md` is the free-text journal.
- **Per-review directory layout.** Each registered review gets `pm/docs/adversarial-review/reviews/<id>/` containing `STATE.md`, `UI_FOCUS.md`, `NOTES.md`, and per-cycle files. Pre-existing top-level audit/cycle files stay as historical archives â€” no backwards compatibility for the audit-browse view.

## Constraints

- No Python iteration runner; the Claude session runs the cycle via its normal tool use.
- No persistent database; markdown is canonical.
- Single user, local-only â€” no multi-user collaboration.
- Sub-100ms walker reactivity to filesystem changes (via `watchdog` + SSE).
- `pm/project.yaml` writes must round-trip-preserve other top-level keys (`plans`, `prs`).
- Walker modifications allowed **only** in the `awaiting-human-review` phase of the current cycle. Every other state renders read-only badges.

## Phases & lock states

| Phase | Breadcrumb (current cycle) | Human can |
|---|---|---|
| `review` | `Cycle N Â· review in progress` | Wait or browse prior cycles |
| `audit` | `Cycle N Â· audit loop running Â· K citations audited` | Wait or browse prior cycles |
| `response` | `Cycle N Â· response in progress` | Wait or browse prior cycles |
| `awaiting-human-review` | `Cycle N Â· ready for your review Â· editable` | Walk changes, click **Apply** |
| `applying` | `Cycle N Â· applying accepted changes` | Wait or browse prior cycles |
| `complete` | `Cycle N Â· complete Â· read-only` | Browse history |

Prior cycles are always read-only.

## Server-pushed updates

Walker server uses `watchdog` to watch per-review `STATE.md`, `UI_FOCUS.md`, and the current cycle's `REVIEW_RESPONSE_CYCLE_N.md`. Filesystem events trigger SSE pushes on `/events?review=<id>`. No polling. Session watches `STATE.md` symmetrically for the Apply transition.

## Non-goals

- Python iteration runner.
- Auto-run as a CLI flag on a Python command (auto-run is the human telling the session to loop).
- Multi-user collaboration.
- Persistent database.
- Backwards compatibility with pre-flow `CITATION_AUDIT_*.md` files.
