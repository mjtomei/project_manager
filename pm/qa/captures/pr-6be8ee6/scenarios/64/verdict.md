# Scenario 64: End-to-end bug-fix session produces pre-fix and post-fix captures

INPUT_REQUIRED

refiner rejected: This scenario can't be driven from a user-facing surface — the "5-step bug-fix flow" is a prompt block appended to impl prompts (`pm_core/bug_fix_prompts.py`), not a command. There is no `pm bug-fix session` entry point; producing the asserted artifacts (`impl/pre-fix/` and `impl/post-fix/` with real manifests + recordings showing the bug reproducing, plus a committed failing-then-passing test) requires the test agent to itself play the role of the impl worker — manually reproing a bug, writing a test, applying a fix, and capturing tmux recordings — which is a multi-hour judgment-driven workflow rather than a deterministic surface action. The follow-up QA leg compounds this by requiring a real scenario container to run to completion against the same PR. The draft also assumes stale capture paths (`pm/qa/captures/<pr-id>/...`); the real layout is `~/.pm/sessions/<tag>/captures/<pr-id>/impl/{pre,post}-fix/` resolved via `pm qa captures-path`. The deterministic pieces this scenario implicitly relies on (bug-fix prompt block contents, captures-path resolution, container bind-mount at `/pm-captures`, QA scenario capture layout) are already covered by narrower scenarios and unit tests; bundling them behind a human-in-the-loop bug-fix run is what makes this one unrunnable.
