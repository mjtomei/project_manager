# QA: End-to-end regression-loop review (pr-d60d185)

This document is the QA checklist for the autonomous regression loop. It
covers two passes:

1. **Static review** of the three integration paths flagged during pr-e58459b
   that unit tests could not fully cover. These are confirmed by code reading
   in this PR and re-verified at runtime by the bug-fix watcher (pr-e3a711c)
   exercising them naturally during the live pass.
2. **Live multi-hour pass** of the full regression loop that the operator
   runs by following the scenarios below. Driven by `INPUT_REQUIRED` so the
   PR session can re-engage Claude over a long observation window.

## Auto-start command under test

```
pm watcher start regression-loop
```

Starts `discovery` + `bug-fix-impl` + `improvement-fix-impl` together
(implementation: `pm_core/cli/watcher.py:_run_regression_loop`). Per-watcher
start (`pm watcher start <type>`) remains available.

`pm watcher list` documents `regression-loop` alongside the three concrete
types.

## Static review of pr-e58459b integration points

### (a) `O` keypress + `_stop_before_merge` short-circuit

- TUI registers the PR in `app._stop_before_merge` when auto-sequence is
  armed via the `O` keybinding (`pm_core/tui/auto_start.py:353`).
- `_maybe_auto_merge` (`pm_core/tui/review_loop_ui.py:453`) checks the set
  and short-circuits with a "ready to merge — press 'g' to merge" log, no
  `pm pr merge` invocation.
- Verified by reading: the set is consulted *after* the auto-start enabled
  + dependency-tree gates, so a non-armed PR still merges normally; an
  armed PR is held even when on the auto-start path.
- Live verification: bug-fix watcher must NOT auto-merge any PR placed
  into `_stop_before_merge`. Live pass will exercise this by arming a
  bug PR with `O`, letting QA pass, then confirming watcher logs
  `ready_to_merge` but does not run `pm pr merge`.

  **Status:** PENDING live observation — checklist below.

### (b) `_check_impl_idle` against real `PaneIdleTracker`

- `_check_impl_idle` (`pm_core/cli/pr.py:2650`) is invoked by
  auto-sequence (`pm_core/cli/pr.py:2735`) from a *fresh* CLI process
  spawned by the bug-fix watcher's `pm pr auto-sequence <id>` shell-out.
- Concern from pr-e58459b notes: session-keyed read-once `idle_prompt`
  events in `hook_events` could make the transient CLI falsely report
  "running" because the event was already consumed by the long-lived TUI.
- Reviewed: `_check_impl_idle` resolves `session_id_from_transcript` and
  reads `hook_events` directly; the read-once semantics are scoped per
  *session id*, not per process, so a fresh CLI invocation against the
  same session sees the same events.
- Live verification: while a bug-fix watcher tick runs auto-sequence on a
  PR whose impl session is idle, confirm the watcher logs
  `awaiting_review` (proceeds past idle gate) rather than `running`.

  **Status:** PENDING live observation.

### (c) `_launch_qa_detached` from a fresh detached process

- `_launch_qa_detached` (`pm_core/cli/pr.py:2868`) spawns `run_qa_sync`
  in a detached process; the watcher invokes auto-sequence from outside
  the TUI thread, so signal handlers + `pm_session` env vars must be set
  by the detached process itself.
- Reviewed: the function uses `subprocess.Popen` with `start_new_session`
  and resolves `pm_session` from current environment; signal handlers in
  `run_qa_sync` are installed at module import / function entry, not
  inherited from a TUI thread.
- Live verification: QA launched by watcher must produce a transcript
  symlink, finalize on completion, and emit a verdict line consumed by
  the next bug-fix-impl tick.

  **Status:** PENDING live observation.

## Live multi-hour scenarios

Operator runs these in a fresh project after merging pr-e3a711c +
pr-d39a7fb + pr-271cb3a + pr-e84b43c. Each scenario uses
`pm watcher start regression-loop` from a tmux pane.

### Scenario 1: bug cycle (auto-merge on PASS)

1. Seed a regression test that fails on a known bug in the project.
2. Watch `discovery` schedule the regression test → test session files a
   bug into `bugs.md`.
3. Watch `discovery` reconcile/dedup on the next tick.
4. Watch `bug-fix-impl` pick up the bug PR → run `pm pr auto-sequence` →
   start/review/QA progress.
5. On QA PASS, confirm `bug-fix-impl` runs `pm pr merge` automatically.
6. Confirm the merged commit lands on master and the regression test
   passes on the next discovery tick.

**Pass criteria:** end-to-end cycle completes without operator
intervention; transcripts and `pm/watchers/bug-fix-impl.log` show the
expected stages.

### Scenario 2: improvement cycle (gated merge)

1. File an improvement into `improvements.md` (manually or via discovery).
2. Watch `improvement-fix-impl` pick it up → auto-sequence runs
   start/review/QA.
3. On QA PASS, confirm watcher leaves the PR in `ready_to_merge` state
   and **does not** auto-merge. Watcher emits `READY` for that tick.
4. Operator runs `pm pr merge <id>` manually; confirm it merges cleanly.

**Pass criteria:** improvement watcher never invokes `pm pr merge`; the
human-merge wait state is observable in the work log.

### Scenario 3: armed bug PR via `O` keypress

1. While the bug cycle is mid-flight, press `O` in the TUI on the bug PR.
2. Confirm the PR enters `_stop_before_merge`.
3. Let QA pass.
4. Confirm `bug-fix-impl` watcher reports `ready_to_merge` but does NOT
   merge; TUI shows the green "press 'g' to merge" log line.
5. Press `g`; confirm merge proceeds.

**Pass criteria:** `_stop_before_merge` gate holds against the watcher's
auto-merge path. (Covers integration point (a).)

### Scenario 4: input-required escalation

1. Cause a bug-fix to repeatedly fail QA (e.g. by misnaming the test).
2. Confirm `bug-fix-impl` eventually emits `INPUT_REQUIRED` per the
   stuck-PR detection in the prompt (pr-e3a711c).
3. Confirm the TUI sticky log surfaces the watcher's INPUT_REQUIRED.

**Pass criteria:** stuck loops escalate to a human rather than spinning
indefinitely.
