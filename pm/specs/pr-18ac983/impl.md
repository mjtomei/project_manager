# Spec: Session Health Watcher (pr-18ac983)

## Requirements (grounded in code)

1. **New built-in watcher** that uses the BaseWatcher framework
   (`pm_core/watcher_base.py:85`) following the pattern of
   `pm_core/watchers/auto_start_watcher.py` and `discovery_supervisor.py`.
   - File: `pm_core/watchers/session_health_watcher.py`
   - `WATCHER_TYPE = "session-health"`, `DISPLAY_NAME = "Session Health Watcher"`,
     `WINDOW_NAME = "session-health"`, `DEFAULT_INTERVAL = 60` (sessions can stall
     quickly; a tighter interval than the 1800s discovery default makes sense).
   - `VERDICTS = ("READY", "INPUT_REQUIRED")`.

2. **Register the watcher** in `pm_core/watchers/__init__.py`'s
   `WATCHER_REGISTRY` so `pm watcher start session-health`,
   `pm watcher list`, and `WatcherManager` discovery work.

3. **Per-tick prompt** generated by a new
   `generate_session_health_prompt(data, session_name, iteration, loop_id,
   meta_pm_root)` in `pm_core/prompt_gen.py`, mirroring the structure of
   `generate_bug_fix_impl_prompt` (line 1058) and
   `generate_discovery_supervisor_prompt` (line 914). Includes:
   - `tui_section(session_name)` block.
   - `notes.notes_for_prompt(root, "watcher")` for general + watcher-specific notes.
   - Persistent work log at `{meta_pm_root}/watchers/session-health.log` with the
     same `mkdir -p` / `tail -n 40` bootstrap used by the other watchers.

4. **Internal launch dispatch** in
   `pm_core/cli/watcher.py::_create_watcher_window` (line 311): add a
   `watcher_type == "session-health"` branch that calls the new prompt
   generator (mirrors the `discovery` branch at line 338).

5. **Detection — what the prompt instructs Claude to look for** in each
   non-watcher pane of the pm tmux session:
   - API errors / HTTP 500s in pane tail (`tmux capture-pane -p`).
   - Claude usage-limit messages with retry-after timestamps (parse the
     ISO/epoch time and skip recovery for that pane until the time elapses).
   - OOM kills (process gone + dmesg/journal evidence, or pane shows
     "Killed"/segfault). Subsumes pr-72f1278.
   - Output stalls: pane has produced no new bytes for ≥ stall threshold
     (cross-check against `pane_idle` hook events — a pane that emitted
     `idle_prompt` is *expected* to be quiet, not stuck).

6. **Recovery actions** the prompt authorizes:
   - Nudge: send Enter or a brief retry prompt via `tmux send-keys` for a
     transient API error / 500 (subsumes pr-76fa48d).
   - Restart: kill the pane and re-launch via the same auto-sequence /
     restart command the PR was using (`pm pr auto-sequence <id>` for
     review/QA, `pm pr restart <id>` for impl).
   - Wait: for usage-limit retry-after timestamps, do nothing this tick;
     log "waiting until {ts}" and re-check next tick.

7. **Retry-count limits**: per pane / pr_id retry count is read from the
   work log. Threshold (proposed: 3 consecutive recoveries) → emit
   `INPUT_REQUIRED` instead of attempting another recovery.

8. **Logging**: every detection and every recovery action appended as one
   ISO-timestamped line to `{meta_pm_root}/watchers/session-health.log`,
   matching the style at prompt_gen.py:1205.

## Implicit Requirements

- The watcher's own pane (window `session-health`) and the watcher
  windows of other watchers (`watcher`, `discovery`, `bug-fix-impl`,
  `improvement-fix-impl`) **must be excluded** from health checks — the
  prompt explicitly lists these as skip targets so the watcher does not
  recover itself.
- The prompt must be self-contained for an unattended Claude tick:
  enumerate panes via `tmux list-windows`/`list-panes`, capture tails via
  `tmux capture-pane`, write the log with shell. No new pm CLI needed.
- The watcher loop engine already handles INPUT_REQUIRED escalation and
  follow-up verdict polling (`watcher_base.py:326`); the new watcher
  inherits this for free.
- `parse_verdict` keeps the same last-line `match_verdict` scan as
  AutoStartWatcher (defaults to `READY` on no match) — using
  `loop_shared.match_verdict`.

## Ambiguities (resolved)

- **DEFAULT_INTERVAL**: choosing 60s. The watcher tick is cheap (read
  panes, write log) and stuck-session recovery has user-visible value in
  seconds-to-minutes, not 30-minute discovery cadence.
- **Stall threshold**: proposed 5 minutes of no new pane output AND no
  `idle_prompt` hook event. Encoded in the prompt text, not as a
  constant — keeps tuning textual.
- **Retry-count threshold**: 3 consecutive recoveries on the same target.
- **Where retry counts live**: in the work log (the prompt scans the last
  ~40 lines and counts entries for the same pr_id) — same pattern as
  bug-fix-impl's "stuck PR" detection at prompt_gen.py:1183.
- **Exclusions**: the prompt skips watcher windows and the user's main
  window (window 0). Implementation-session panes / review-loop panes /
  QA panes are the targets.

## Edge Cases

- **Pane disappears mid-tick** (`pane_exists` false): treat as already-
  recovered or user-killed; log + skip, do not relaunch unless the
  governing pr_id is in a status that expects an active pane.
- **Multiple watchers running simultaneously**: this watcher must not
  touch other watcher panes (covered by the skip list).
- **Usage-limit retry-after in the past**: treat as expired; eligible to
  retry this tick.
- **No Claude panes at all** (empty session): emit READY immediately.
- **The watcher pane itself stalls**: out of scope — handled by the
  outer pm TUI watchdog (#113) that supervises the watcher manager.
