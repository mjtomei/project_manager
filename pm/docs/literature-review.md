# Literature Review: Autonomous Regression and Bug-Fix Loop

## Introduction

This review surveys the work behind a plan that extends a regression-test suite so it also notices what it is not catching and files new work for itself when it does. (A regression-test suite is the standing set of automated checks a project runs to confirm that yesterday's fix did not break today's feature.) The plan lives at `pm/plans/plan-regression.md` and is implemented in the `pm` project-manager tool.

**What the plan does, in plain English.** The plan installs three background processes ("watchers") that share custody of a project's quality. The first watches for new bugs and coverage gaps by running the regression tests on a schedule. The other two drive the resulting work — one for bug fixes, one for user-experience improvements — through a reproduce-fix-verify cycle, with a separate review step that grades each candidate change.

Two evaluators close the loop. A "scenario quality supervisor" checks that the review step itself was thorough enough before a change is allowed to merge. A capstone evaluation closes the loop: a single-prompt task with real internet access, followed by an audit that walks the agent's transcript to verify the agent did the work rather than searching for the answer online.

Watcher hierarchy at a glance:

```
external trigger (cron / keypress / "wake up")
        │
        ▼
discovery supervisor ──────────► files PR (bug or improvement)
   reads regression results              │
                                         ▼
                                 implementation watcher
                                 (one for bugs, one for UX)
                                   │  reproduce → fix → verify
                                   ▼
                                 QA review loop
                                   │  runs the regression, watches, grades
                                   ▼
                                 scenario quality supervisor
                                   │  was the QA itself thorough enough?
                                   ▼
                                 merge gate
```

Each arrow carries different information: the discovery supervisor passes a bug or improvement description; the implementation watcher passes a candidate diff (a line-by-line listing of the proposed code change); the QA review loop passes a verdict plus captured evidence; the supervisor passes a quality verdict on the QA work itself; the merge gate decides whether the change ships.

**Terms used throughout — defined in the Appendix.** The vocabulary the review leans on most: *LLM, Claude, Claude Code, pull request (PR), merge, CI, QA, UX, watcher, the loop, implementation watcher, discovery supervisor, QA review loop, capstone, auto-sequence, regression-test suite.* Each is glossed inline at first body use; the full Appendix entries are at the end of the document.

That design crosses several research areas, and the contamination question runs through all of them in two forms. The familiar form is *pretraining contamination* — has the benchmark already appeared in the LLM's training data? The form the capstone confronts is *runtime contamination* — during the evaluation itself, did the agent fetch the reference solution off the open internet? These are different problems with different defenses, and the capstone bets on the second.

This review walks through the surrounding work in roughly plan order: what to evaluate against (Section 1), what to build with (Section 2), how to test (Section 3), automated program repair (Section 4), how to keep the loop honest (Sections 5 and 6), and how others have organized continuous-quality bots (Section 7). The goal is to give a reader enough context to know which prior work the design is leaning on, which ideas it borrows piecemeal, and where it deliberately departs.

Sections cite the relevant PR ids from the plan so the reader can move between review and plan without losing place.

## 1. Agentic Coding Benchmarks

If you want to know whether a coding assistant is any good, you need a test it has not already seen the answer to. The history of those tests has three eras. The plan's capstone evaluation belongs in the third.

The first generation, **HumanEval** (Chen et al. 2021, the Codex technical report; Codex was the LLM that powered the first version of GitHub Copilot) and **MBPP** (Austin et al. 2021), evaluated a model on isolated function-completion problems: write a single function to match a docstring. Grading came from hidden tests — small automated checks (called unit tests) that run a function and confirm it produced the right output. "Hidden" means the candidate doesn't see them, so they can't be gamed. HumanEval and MBPP were the right benchmarks for their era. They reveal almost nothing about a system that has to live inside a real codebase. They were also rapidly contaminated: by the time GPT-4 arrived, both benchmarks had appeared in enough scraped training corpora that scores were difficult to interpret.

The follow-up **EvalPlus** work (Liu, Xia, Wang, Zhang 2023, NeurIPS 2023) made the test-insufficiency problem explicit. It extended HumanEval's tests roughly 80x and showed that pass@k — a scoring metric: give the model `k` tries; count a pass if any try works (pass@1 means one try, pass@10 means ten; `k` is a hyperparameter chosen by whoever runs the benchmark) — dropped by 19 to 29 percent under the stronger suite. That is, scores fell by roughly a fifth to nearly a third when the tests were tightened; a sizeable fraction of code that "passed" the original benchmark was actually wrong. "The test passed" is a function of how good the test is, not just whether the code works.

The second generation responded to the realism gap. **SWE-Bench** (Jimenez, Yang, Wettig, Yao, Pei, Press, Narasimhan 2024, ICLR 2024) collects 2,294 task instances from real Python GitHub repositories — GitHub is the dominant online host for shared code projects, and a "repository" is one such project. Each SWE-Bench task is: given this issue and this codebase, produce a patch (a diff — a line-by-line listing of what was added, removed, or changed in the file) that resolves the issue and passes the hidden test. This is a much harder task and a more realistic one. The associated **SWE-agent** paper (Yang et al. 2024, NeurIPS 2024) argues that what matters is the **agent-computer interface** — ACI for short, used throughout the rest of this review — the menu of commands the agent has available: file-open, file-edit, run-tests, search-codebase. The argument is that what commands you expose to the model matters as much as which model you use. **SWE-Bench Verified** is a 500-task human-filtered subset that addresses some quality issues in the original. **Scale AI's SWE-Bench Pro** (2025) extends the same shape to longer-horizon, multi-file tasks. **SWE-Bench Multimodal** (Yang, Jimenez, Zhang et al. 2024, ICLR 2025) extends the shape into 617 visual-software JavaScript tasks and shows that top SWE-Bench systems struggle on visual problem-solving — orthogonal to this plan but a useful counterweight to "SWE-Bench means Python only."

The third generation is responding to a problem the second never solved: contamination. SWE-Bench tasks are real GitHub issues, which means the resolutions are also on GitHub and likely in training data. The contamination concern became concrete in February 2026, when OpenAI audited the 138 hardest remaining tasks in SWE-Bench Verified and found at least 59.4 percent materially flawed — 49 over-narrow (tests rejected functionally-correct solutions) plus 26 under-specified (tasks required undisclosed extra features) ("Why SWE-bench Verified no longer measures frontier coding capabilities," openai.com, 2026-02-23). For context, 59.4 percent is more than half of the audited subset, not a marginal-edge finding — and frontier models (GPT-5.2, Claude Opus 4.5, Gemini 3 Flash Preview) reproduced gold patches verbatim from the task ID alone. OpenAI recommended Scale AI's **SWE-Bench Pro** as the replacement going forward. The lit review reads the rest of §1 in light of that recommendation, while noting the audit's scope was the hardest 138 tasks rather than the full 500-task Verified set. **LiveCodeBench** (Jain et al. 2024, ICLR 2025) addresses contamination by tying problems to release dates after the model's training cutoff. **BigCodeBench** (Zhuo et al. 2024) emphasizes complex, library-rich function calls that resist simple memorization. A parallel mutation approach — Garg, Steenhoek, Huang's "Saving SWE-Bench" (Microsoft Research, arXiv:2510.08996, v4 2026-01-23) — converts formal GitHub issue descriptions into chat-style user queries and reports that public benchmarks overestimate agent capabilities by over 50 percent on some models when measured against more realistic user phrasings. **ProgramBench** (the 200-task benchmark where Claude Opus 4.7 with mini-swe-agent currently tops the leaderboard — programbench.com snapshot 2026-05-14) takes the radical step of giving the agent only a compiled executable plus user-facing documentation and asking it to *rebuild* the codebase from scratch. ProgramBench runs offline-only inside a sandbox (an isolated environment that lets the agent reach out to the rest of the computer or the internet only when explicitly allowed), with no decompiler and execute-only permissions on the binary.

A caveat on ProgramBench: it is much newer than SWE-Bench and its evidence base is correspondingly thinner. It is cited here based on its website (programbench.com, snapshot 2026-05-14) and an arXiv preprint (2605.03546). It does not yet have the follow-up literature SWE-Bench has accumulated, and the reader should weight claims about ProgramBench's design accordingly.

Rather than tabulate every property of each benchmark, the differentiating story is two-dimensional: *how realistic is the task* (column) and *what does the benchmark do to keep the agent from looking up the answer* (row). Reading the rows top to bottom: defenses get stronger from no defense at all to post-hoc detection.

```

                          completion   on real repo     from binary
```

The post-run detection row is now a populated quadrant, not an empty one: AISI Inspect Scout and NIST CAISI publish transcript-review pipelines for catching cheating modes, Meerkat audits across many traces at once, and AuditBench documents the tool-to-agent gap in alignment-auditing investigator setups. The capstone is not alone in the cell. Its distinctness is *which threat* it targets: it applies the post-run audit shape to runtime-internet lookup of forbidden materials during autonomous agent runs, whereas the precedents target alignment-auditing of hidden behaviors (AuditBench) or transcript-review for known cheating modes on cybersecurity-task agents (NIST CAISI, AISI Inspect Scout, Meerkat). Same architecture, different threat.

The bottom two rows are categorically different defenses, not weaker-and-stronger versions of the same one. ProgramBench *denies* internet access at runtime (cleanroom); the capstone (`pr-e2b7fdf`) *grants* internet access and detects misuse afterward (allowlist audit). (ProgramBench's evidence base is still thinner than SWE-Bench's, as noted above; the contrast is methodological, not a claim that ProgramBench has accumulated SWE-Bench's level of follow-up validation.) These are opposite methodological bets. The capstone's argument is that real engineers use the internet, so an evaluation that bans it is measuring an unrealistic situation. The capstone instead uses a written directive plus a curated `materials/` directory, an allowlist (a written list of which external sources the agent is permitted to consult), and a *post-run integrity audit* that walks the agent's tool-use transcript — the full log of every command the agent ran, every file it read, every web page it fetched, in order — to verify the agent did not look up the answer. The offshoot PR (`pr-0cf3626`) keeps the offline ProgramBench shape for leaderboard purposes.

## 2. Autonomous Coding Agents

The agents that write code under the hood of tools like Aider and Claude Code all derive from a handful of designs. Knowing which ones the plan borrows from tells you where to look for the plan's weaknesses.

**SWE-agent** (Yang et al. 2024) is partly an agent and partly a paper about how to design agents. Its main contribution is the ACI argument — that a small set of carefully chosen commands (file viewer with line numbers, scoped edits, a search command, a test runner) outperforms giving the model raw shell access. Yang et al. effectively defined the modern coding-agent paradigm, and the plan's `BaseWatcher` prompt-builder lineage is downstream of that paradigm regardless of whether the connection was made explicit in the original design notes.

**OpenHands** (Wang et al. 2024, "OpenHands: An Open Platform for AI Software Developers as Generalist Agents," formerly OpenDevin, ICLR 2025) is the open-source platform that has become the de facto baseline on the SWE-Bench Verified leaderboard (swebench.com/verified.html, snapshot 2026-05-14). Its architecture is an event-stream design: every action between agent and environment is logged as a typed event in order, like a chat transcript that also captures the agent's tool calls. The 2025 OpenHands SDK paper (arXiv:2511.03690) refactors this into a stateless, event-sourced shape with separate SDK, Tools, Workspace, and Server packages — and reports a 72 percent resolve rate on SWE-Bench Verified with Claude Sonnet 4.5 plus extended thinking. That is roughly seven out of every ten SWE-Bench Verified tasks resolved. OpenHands supports hierarchical agents that can delegate to other agents, which is the literature's nearest match to the plan's three-watcher structure (Section 7).

**Devin** (Cognition Labs, the startup that introduced Devin in a 2024 blog post; no peer-reviewed paper) made the splash that put autonomous coding agents in the broader software-engineering conversation under a "junior engineer" framing. Public technical content remains thin — Cognition has published blog posts and a 2025 "performance review" but no architecture paper. Architectural details must be inferred from the product surface, which offers a sandboxed environment with shell, editor, and browser; long-horizon planning over thousands of decisions; an emphasis on parallel runs.

**mini-swe-agent** is the minimal scaffold (the surrounding code that gives a model its commands, file access, and step-by-step structure; the LLM itself is the engine, the scaffold is the chassis) that ProgramBench uses as its baseline. The rationale is that less harness means less confound between model capability and tool design. **Aider, Cline, Claude Code, and OpenAI's Codex CLI** are the practitioner-facing tools — command-line tools that let a human chat with an LLM to edit a project, the practitioner-facing analogs to one another. Informal practitioner comparisons exist; see the References for pointers. None of these are academic.

The plan's iterative-loop design is not the only credible approach. **Agentless** (Xia, Deng, Dunn, Zhang 2024, arXiv:2407.01489) deliberately removes the agent loop entirely: a three-phase pipeline of localization, repair, and patch validation with no iterative tool use, no scratchpad, no agentic plan revision. It beat many agent-based systems on SWE-Bench Lite — roughly one out of every three real GitHub issues successfully fixed (32.0 percent) at $0.70 per task in the original paper, about the cost of a cup of coffee per bug fix, when SWE-agent at the time was several dollars per task — and is the explicit counter-design to the SWE-agent / OpenHands lineage. **AutoCodeRover** (Zhang, Ruan, Fan, Roychoudhury 2024, ISSTA 2024, arXiv:2404.05427) takes a different tack: AST-based program analysis grounds the LLM's code context. Instead of letting the LLM browse the code freely, the system uses the program's structural tree to look up exactly where a bug lives, then hands that location to the LLM — like consulting a building's blueprint to find a leaky pipe instead of wandering room to room. ("AST," abstract syntax tree, is a tree-shaped map of a program's structure, the way a sentence-diagram is a tree-shaped map of a sentence.) Both papers are evidence that the design space is broader than the SWE-agent / OpenHands lineage. The plan's commitment to a loop-based architecture should be understood as a deliberate bet, not a default.

Because the plan reuses Claude Code as the underlying agent for every watcher tick, the architectural question is not "how do we build an agent?" but "how do we shape its surface?" The plan's `BaseWatcher` (`pr-3032fb6`, already merged) and prompt-builder pattern is closest in spirit to SWE-agent's ACI argument: don't change the model, design the surface carefully. The watcher prompts pass curated context (work logs, plan state, user notes) and constrain the verdict surface (the limited set of grades the system can emit — in `pm`, just `READY` or `INPUT_REQUIRED`).

## 3. LLM-Driven Test Generation and Fuzzing

When the plan's regression tests miss something, the tool tries to write a new test on the fly. There's a decade of research on whether that ever works.

*This section covers two pieces: §3.1 on LLM-driven test generation, and §3.2 on test doubles for LLM-using code. Automated program repair — the parallel problem of fixing bugs once a failing test is in hand — has its own section (§4).*

### 3.1 LLM-driven test generation

The QA scenario planner in Phase 10 (`pr-2680fbf`) authors a brand-new regression test on the fly when no existing one fits. That puts the plan squarely in the LLM-test-generation literature.

**CodaMosa** (Lemieux, Inala, Lahiri, Sen 2023, ICSE 2023) is the seminal paper. It combines evolutionary test generation (an older AI technique that breeds candidate test cases like genes, keeping the ones that perform best and mutating them to make the next generation; the search-based testing tradition, of which **Pynguin** is the leading classical Python implementation) with Codex: when the evolutionary search hits a coverage plateau (coverage being the percentage of the program's lines or branches the test suite actually exercises), CodaMosa asks the LLM for fresh test cases targeting uncovered code. The hybrid significantly outperforms either component alone. The deeper pattern — *use the LLM as the escape valve when a classical algorithm gets stuck* — is more portable than CodaMosa itself. The plan's coverage-driven scenario growth (`pr-c2397e2`) is an instance: when coverage thresholds aren't met, the planner is asked to draft scenarios targeting the uncovered surface.

The wave of **ChatGPT-as-test-generator** papers followed. **ChatTester** (Yuan et al. 2023, FSE 2024) is one of the first systematic empirical studies. **ChatUniTest** (Xie et al. 2023) introduces a Generation-Validation-Repair (GVR) loop that catches and fixes compile errors and assertion failures before returning the test, a pattern the plan's QA scenario quality supervisor (`pr-98f670e`) directly mirrors. **TestPilot** (Schäfer, Nadi, Eghbali, Tip 2023, IEEE TSE 2024) is the GitHub Next group's (GitHub's research arm) npm-focused (npm is the standard package system for JavaScript code) test-generation tool — a clean baseline for prompt-skeleton-based test generation without any LLM fine-tuning. EvalPlus (cited in §1) is the empirical sister: even when an LLM does generate plausible-looking tests, the harder problem is whether the tests are actually *sufficient*. A follow-up empirical literature (Siddiq et al. 2024; arXiv:2406.18181 surveys it) shows that out-of-the-box LLM test generation has nontrivial failure modes — flaky assertions (checks that sometimes pass and sometimes fail for the same input), missing edge cases, tests that replace so many real components with stand-ins that they don't really exercise the code under test — and that pairing the LLM with a coverage-driven loop or a search-based component is the consistent winner.

**Mündler et al. 2024 "SWT-Bench"** (arXiv:2406.12952) is the most direct missing peer: it evaluates LLMs at generating tests rather than fixing bugs, on real-world GitHub bug-fix tasks. SWT-Bench is the published benchmark `pr-2680fbf`'s scenario-authoring flow most resembles — produce a failing test from a natural-language description of a defect. **Jain et al. 2024 R2E** (ICML 2024) does repository-level test generation with executable feedback: it extracts functions from a GitHub repo, generates equivalence tests using LLMs with program analysis, and feeds failing-test feedback back to the agent. R2E is the closest published architecture to a regression-authoring loop that uses execution as the verifier; the plan's `pr-2680fbf` and `pr-51586d2` (mocks awareness in regression authoring) lift the same pattern to the regression-suite level.

The 2025-2026 wave on the SWT-Bench shape supplies four direct prior peers. **Issue2Test** (Mündler, Bouzid, Müller, Vechev 2025, arXiv:2503.16320) generates issue-reproducing test cases from GitHub issue reports, evaluated on SWT-Bench-lite; it reports generating tests for 91 instances (13 more than the strongest baseline) at roughly 5 cents per issue. **SWE-Tester** (arXiv:2601.13713, 2026-01) trains open-source LLMs on 41,000 issue-reproduction instances and reports absolute gains up to 10 percent in success rate and 21 percent in change coverage on SWT-Bench Verified — the training-side counterpart to Issue2Test's prompt-engineering pipeline. **Otter / e-Otter++** (arXiv:2502.05368; arXiv:2508.06365) is the heterogeneous-prompting line: Otter uses a self-reflective action planner to decide what to read and what to write; e-Otter adds execution feedback for repair; e-Otter++ adds inference scaling and reports an average 63 percent fail-to-pass rate on TDD-Bench Verified. **TestExplora** (arXiv:2602.10471, Microsoft Research 2026) is the most structurally relevant peer to `pr-2680fbf` because it tests the proactive case — 2,389 tasks across 482 repositories with all defect-related signals hidden, so the model has to discover latent defects by comparing the implementation against documentation-derived intent rather than reading a defect report. TestExplora's headline finding is that state-of-the-art LLMs reach only a 16.06 percent fail-to-pass rate on the proactive task, well below the reactive-reproduction numbers above. Issue2Test, SWE-Tester, and Otter all operate in the reactive shape (an issue report exists); TestExplora operates in the proactive shape (no defect signal). The `pr-2680fbf` setup — planner authors a new regression when no existing one fits — is the proactive shape, so TestExplora is the closest peer, and the plan should narrow its novelty claim against that benchmark specifically: `pr-2680fbf` differs from TestExplora in that (a) it persists the drafted regression to the corpus rather than scoring once, (b) it operates inside a continuous watcher loop with coverage feedback rather than a one-shot benchmark, and (c) it consults the discovery supervisor's running work log for context, not a fixed documentation oracle. The 16.06 percent TestExplora ceiling is the strongest published prior-art number against which `pr-2680fbf`'s drafted-regression yield should eventually be measured.

**Fuzz4All** (Xia, Paltenghi, Tian, Pradel, Zhang 2024, ICSE 2024) is the most useful adjacent paper. It uses an LLM as a fuzzer's input generator across six languages (fuzzing means automatically generating large numbers of weird or random inputs to a program to see what breaks) and shows that an *autoprompting* loop — where the LLM iteratively proposes inputs, observes coverage, and adjusts its own prompt to target uncovered surface — beats hand-built language-specific fuzzers on coverage. Fuzz4All doesn't stop at "LLM generates an input." It feeds coverage measurements back into prompt construction so the next batch of inputs targets uncovered regions. The plan's coverage-driven scenario growth (`pr-c2397e2`) applies the same coverage-feedback pattern at a coarser grain. The 2024 LLM-based fuzzing survey (arXiv:2402.00350) catalogs the broader space.

Where the plan departs from CodaMosa and Fuzz4All is on *durability*. Both produce throwaway suites for that run. The plan's `pr-2680fbf` commits the drafted regression test as a markdown file in `pm/qa/regression/` and registers it for the discovery supervisor to rerun on its own schedule. Each QA run that hits a new surface deposits a durable artifact. The nearest analog is not a paper but the idea behind OSS-Fuzz (Serebryany, USENIX Security 2017) — Google's continuous fuzzing service, where a corpus grows over time and survives the run that generated it. The discipline OSS-Fuzz illustrates is that "durable" earns its keep when measured: bugs found per week, corpus size growth. The plan inherits the design pattern. Whether it also inherits the measurement discipline — and whether durable tests *compound* rather than merely accumulate — is the question the Conclusion takes up.

### 3.2 A note on test doubles for LLM-using code

Testing code that calls an LLM is hard because every call costs money and gives slightly different answers. This subsection is about how the plan fakes the LLM during its own tests.

The plan includes fakes infrastructure (Phase 6's bridge `pr-fbda1a8`, Phase 10's `pr-abcf70f` `FakeClaudeSession` and `pr-9603d04` `FakeGitHubBackend`). These are test doubles — stand-ins for a real component used during testing, so the test can run without the real thing being available. ("Mock" is a strict stand-in that fails the test if called the wrong way; "fake" is a lightweight working substitute.) A redone search for academic peers — "LLM mock," "record-replay LLM testing," "deterministic LLM unit test," "fake GitHub backend test" — surfaced nothing peer-reviewed. The honest negative result: this corner of the design space is industry-tooling territory (LiteLLM's mock mode, the VCR.py / `pytest-recording` record-replay pattern, LangWatch's Scenario framework), and the plan's contribution is applying the classical testing-pattern literature that distinguishes mocks (strict stand-ins) from fakes (working substitutes) to the LLM-and-version-control boundary, in the absence of academic guidance specific to that boundary.

## 4. Automated Program Repair

Once a failing test points at a bug, the next question is whether the tool can fix it on its own. That problem is called automated program repair (APR), and it has been worked on for over a decade.

The bug-fix watcher (`pr-30588a7`, already merged) detects a failing test, proposes a fix, verifies the test passes, and gates on human review. That is automated program repair — a long pre-LLM literature that does almost exactly this job, just with different generation mechanisms.

**GenProg** (Le Goues, Nguyen, Forrest, Weimer 2012, IEEE TSE 2012) is the canonical citation: genetic programming — an older AI technique that breeds candidate program variants like genes — evolves variants until the test suite passes, then minimizes the diff. **Sapfix / Sapienz** (Marginean, Bader, Chandra, Harman, Jia, Mao, Mols, Scott 2019, ICSE-SEIP 2019, "SapFix: Automated End-to-End Repair at Scale") is Facebook's deployment of the same pattern at production scale — automated fault fixing across millions of lines of code, with engineer-approval gates before deployment. **Getafix** (Bader et al. 2019, OOPSLA 2019) extracts repair templates from past human fixes and applies them to new bugs. More recent CodeLlama- and GPT-based repair work continues the tradition. **Chen et al. 2023 "Teaching Large Language Models to Self-Debug"** (arXiv:2304.05128) bridges this literature with the self-refinement work in §5: the model executes a candidate program, observes the result, explains the bug to itself, and tries again. **Zhang et al. 2023 "Self-Edit"** (ACL 2023) does the same in a generate-and-edit shape, using execution feedback on an example test case to guide a code editor.

The plan's bug-fix flow re-runs APR's decade-old playbook with a new generation mechanism. Genetic programming has given way to the LLM, but the orchestration around the generation step is the closest academic analog the design has. `pr-30588a7`'s reproduce-fix-verify cycle is structurally similar to Self-Debug (Chen et al. 2023) but operates at the watcher-tick level — many minutes apart, with a persistent work log between ticks. The longer feedback loop changes the failure modes: drift, stale context, and intervening discovery-supervisor activity replace Self-Debug's faster-but-tighter pathologies.

## 5. Self-Improving and Iterative LLM Loops

When you ask a language model to grade its own work, sometimes it gets better and sometimes it gets worse. The research literature has figured out when each happens, and the plan is designed to land on the "gets better" side.

The pattern of "have the LLM critique its own output and try again" has a sizeable 2023-vintage academic literature, and a sharply skeptical 2023-2024 follow-up literature.

**Self-Refine** (Madaan et al. 2023, NeurIPS 2023) is the canonical citation. The same LLM acts as generator, feedback provider, and refiner, in that order, for some number of iterations. The paper reports roughly 20 percent absolute improvement across seven tasks. **Reflexion** (Shinn, Cassano, Berman, Gopinath, Narasimhan, Yao 2023, NeurIPS 2023) adds a memory buffer: the agent reflects after each trial, the reflection is stored, and subsequent trials are conditioned on the accumulated reflections. The architecture splits into Actor, Evaluator, and Self-Reflection roles. Reflexion's *verbal reinforcement learning* design — instead of retraining the model itself, the system updates a written "lessons learned" note that gets pasted into the next attempt's prompt — is the direct ancestor of the plan's `pm/watchers/*.log` substrate. Each watcher's work log is precisely Reflexion's episodic memory (a written log of past attempts and what was learned from each; the next attempt reads the log before starting), applied across watcher ticks instead of across agent trials.

The sharp pushback came soon after. **Huang et al. 2023, "Large Language Models Cannot Self-Correct Reasoning Yet"** (arXiv:2310.01798, ICLR 2024) is the sharpest rebuttal. When an LLM is given only its prior output and asked to improve it without new information — what Huang et al. call *intrinsic self-correction*, asking the model to look at its own last answer and improve it, with no new data, no test result, no human feedback, just "try again, better" — performance often *degrades*.

Huang et al. are explicit that this finding is bounded: it applies to the intrinsic setting, not to settings where the model receives external feedback (tool calls, ground truth, oracle access — a trusted source the model can ask for the correct answer, like a graded answer key), where self-correction reliably helps.

The plan's loops are not intrinsic self-correction. Each iteration produces new artifacts — regression-test files, evidence captures, tool-use transcripts, coverage reports — that mechanical systems check independently of the model's reasoning. Concretely: the scenario quality supervisor reads the scenario's tool-use transcript, not its reasoning trace (`pr-98f670e`); the bug-fix flow refuses to attempt a fix until a failing test demonstrates the bug (`pr-30588a7`); the capstone audit walks the agent's tool calls against an allowlist (`pr-e2b7fdf`); the coverage gates measure code execution rather than model self-assessment (`pr-b42059d`, `pr-8ed578d`). Huang et al.'s caution still applies, but the bound matters: the plan operates where self-correction-with-external-feedback works, not where it degrades.

The positive-side literature on this setting is **verifier-guided generation**: train or prompt a separate model whose only job is to grade candidates, then pick or refine based on its score. **Cobbe et al. 2021, "Training Verifiers to Solve Math Word Problems"** (arXiv:2110.14168) is the foundational reference: a separately-trained verifier that ranks candidate solutions on math word problems outperforms a finetuned generator alone. **Lightman et al. 2023, "Let's Verify Step by Step"** (NeurIPS 2023, arXiv:2305.20050) extends this to *process supervision* — the verifier scores each intermediate reasoning step, not just the final answer — and shows process-supervised verifiers solve 78 percent of a hard MATH subset where outcome-only supervision lags. The plan's scenario quality supervisor belongs to this sub-genre: a separately-prompted Claude session whose only job is to grade the QA session's evidence depth. Cobbe / Lightman is the positive-side prior art; Pan 2024 (below) is the critique to defend against.

**Pan, He, Bowman, Feng 2024 "Spontaneous Reward Hacking in Iterative Self-Refinement"** (arXiv:2407.04549) is the natural critique. Reward hacking — when a model figures out how to score well on a measurement without actually doing the underlying task, for example modifying the test instead of fixing the bug — emerges spontaneously in iterative self-refinement, more readily when the generator and the evaluator share context. Skalse et al. 2022 ("Defining and Characterizing Reward Hacking," NeurIPS 2022) and Pan, Bhatia, Steinhardt 2022 ("The Effects of Reward Misspecification," ICLR 2022) are the peer-reviewed theoretical backing — Skalse formalizes what "unhackable" can and cannot mean, Pan 2022 shows that more capable agents reliably exploit reward misspecifications more thoroughly. Persona Vectors (Chen, Arditi, Sleight, Evans, Lindsey 2025, arXiv:2507.21509) provides the precedent for treating sycophancy — the model's tendency to agree with whatever the user appears to want rather than to push back when the evidence calls for it — as a model-side direction in activation space — the kind of artifact a future scenario quality supervisor could probe for directly in evaluating autonomous-loop runs, complementary to Pan 2024's behavioral-emergence finding.

An adjacent failure mode the supervisor design has to consider: Anthropic's **AuditBench** (alignment.anthropic.com, 2026-03-10) is a benchmark of 56 fine-tuned models with implanted hidden behaviors (sycophantic deference, opposition to AI regulation, secret loyalties) used to evaluate alignment-auditing investigator agents (an *investigator agent* is an LLM acting as an auditor — given access to a target model's transcript or outputs, asked to identify behaviors of interest) across 13 tool configurations. AuditBench documents a *tool-to-agent gap* — the gap between what an LLM-as-judge tool catches when asked one-shot vs. what an LLM agent catches when given the same evidence and time to investigate: tools that surface accurate evidence in isolation often fail to improve investigator-agent performance, because agents underuse the tools, struggle to distinguish signal from noise, or fail to convert evidence into correct hypotheses. AuditBench's target task is hidden-behavior alignment auditing rather than QA-scenario auditing, so the result does not transfer literally — but the structural concern (a checker reading evidence and aggregating it into a verdict can fail at the aggregation step even when the evidence is good) is shared with the scenario-quality supervisor (`pr-98f670e`). Read narrowly, AuditBench is a reason to expect the supervisor's prompt design — specifically how it weighs and aggregates evidence — to be the under-validated piece, separate from any question about the `--resume` mechanism.

The Pan 2024 finding identifies context-sharing as a factor, not separation as the prescribed mitigation — that distinction matters for the plan's design. The plan's QA scenario quality supervisor (`pr-98f670e`) attempts to reduce shared context by running the supervisor as a separate session that reads the scenario's captures and tool-use transcript via Claude Code's `--resume` mechanism rather than running inside the scenario's loop. What `--resume` actually shares is a material detail. It carries forward the session's accumulated state — system prompt, prior turns, tool-use history. It does *not* run as the same continuous reasoning process, but the context-window content is the scenario's full thread. The amendment cap (default 2) is the primary mitigation; the `--resume` arrangement is a secondary reduction of shared context. Two amendments is exactly the regime Pan et al. found measurable hacking in, so the amendment cap is a necessary but not sufficient defense; the supervisor architecture is the most under-validated mitigation in the plan. A stronger redesign — fresh session with captures-only input bundle, optionally a different model or a K=3 ensemble for merge-gating verdicts — is sketched in `pm/docs/adversarial-review/RESUME_DESIGN_NOTE.md` for readers who want the proposed alternative.

**AutoGen** (Wu et al. 2023, COLM 2024) and **MetaGPT** (Hong et al. 2024, ICLR 2024) are the two most-cited multi-agent frameworks. AutoGen emphasizes flexible conversation patterns between configurable agents and is general-purpose. MetaGPT explicitly assigns software-engineering roles — Product Manager, Architect, Engineer — and runs them as a pipeline. The plan's watchers are closer to MetaGPT's role-based shape than to AutoGen's free-form conversation, but the plan is more constrained: each watcher has one tightly-scoped job, ticks rather than free-form dialogue, and a flat structure rather than the hierarchical orchestration AutoGen supports.

The plan's design at a single-tick level pairs a *writer* (which produces a candidate answer) with a *checker* (which decides whether to accept it, and if not, what to change). Rejected candidates get rewritten. This is called actor-critic in the research literature, but the plain names are clearer. The implementation watcher is the writer, the QA review loop is the checker, and the verdict gate is the merge bar. That pattern is the abstraction the plan most closely fits.

## 6. Agent Integrity, Benchmark Contamination, and No-Cheat Verification

If the plan's evaluation lets the model use the internet, the model could just look up the answer. This section is about whether anyone has figured out how to tell when it does. This is the load-bearing literature for the capstone PR (`pr-e2b7fdf`), and the area where the plan most depends on still-actively-shipping prior work. Two transcript-review pipelines anchor the precedent set — NIST CAISI in the US and AISI Inspect Scout in the UK — with AuditBench and Meerkat adjacent on different problems. The capstone's audit is an integration job on top of that set, not a step into an empty field.

### 6.1 Pretraining contamination

Before you trust a benchmark score, ask whether the answer was already in the model's training data. This subsection is the literature on detecting that.

**Sainz et al.** ("NLP Evaluation in Trouble: On the Need to Measure LLM Data Contamination for Each Benchmark," Findings of EMNLP 2023) defined a taxonomy of contamination levels and made the case that every benchmark should publish an associated contamination measurement. **Riddell, Ni, Cohan 2024** (ACL 2024) applies two ways of detecting near-duplicates that survive renaming: edit-distance (how many character changes would convert one program into the other) and AST-based comparison (matching programs by their structural shape, not their literal text) — checking whether two programs have the same tree-shaped structure even if they were re-worded at the surface. The rationale is that AST structure survives the surface-level rewrites that defeat literal text matching. Applied to popular code benchmarks, the analysis reports significant overlap with open training corpora.

EvalPlus (cited in §1) is a closely related thread: its 80x test extension surfaces "code that passed only because the tests were weak," a different failure mode that rhymes with pretraining contamination — both let inflated scores survive on benchmarks that don't push hard enough.

The 2024-2025 surveys (arXiv:2406.04244, arXiv:2502.14425, arXiv:2502.17521) summarize the post-Sainz wave: per-benchmark measurements are now expected, and dynamic / time-windowed evaluation (the LiveCodeBench approach) is the leading defense.

### 6.2 The threat model the capstone confronts (runtime contamination)

The plan's situation differs structurally. Pretraining contamination asks: was the answer in the training data? The capstone asks a different and harder question: *during the run*, did the agent look up the answer? When internet access is on, the model can fetch the reference implementation, a writeup, or a similar GitHub repository in real time. Pretraining contamination is fixed at training time; runtime contamination is decided turn by turn.

The audit problem decomposes into a structural setup plus three contributions that the §6.3 precedent set (NIST CAISI, AISI Inspect Scout, Meerkat, AuditBench) does not jointly cover. The setup: the sandbox lets the agent reach the open internet (otherwise the test is not realistic); the agent has reason to peek at how others solved the same problem; a single stray click can taint everything that follows; the trace records every action but labels none of them as cheating; and the grader sees only the final answer, where a cheated solution and a legitimate one look identical. The transcript-walk architecture that handles this setup is already published — NIST CAISI and Inspect Scout do single-transcript review, Meerkat does across-runs clustering, AuditBench documents the verdict-aggregation failure mode.

What remains residual to the capstone, given that precedent set, is narrower:

1. **Threat-model match.** The published transcript-audit pipelines target hidden-behavior alignment auditing (AuditBench) or known cheating modes on cybersecurity / CTF agents (NIST CAISI, Inspect Scout). The capstone applies the same architecture to runtime-internet lookup of a reference solution during autonomous coding-agent runs — a different threat with different signals (URLs in tool-use trace vs. behavioral patterns in outputs).
2. **Allowlist-against-trace boundary mechanism.** A written allowlist of permitted external sources, paired with a curated `materials/` directory, against which every fetched URL in the trace is checked. The precedents do not specify this boundary because their threats are not internet-fetching.
3. **Integration into the loop's verdict pathway.** The audit verdict feeds the merge gate of an autonomous continuous-quality loop rather than producing a standalone audit report. None of the §6.3 precedents are situated inside an autonomous loop's merge pathway in this way.

The audit's job is to recover the process signal — walk the transcript after the fact and decide whether any of the agent's external lookups crossed the allowlist boundary. Designing the audit so it does not become its own LLM-on-LLM judge with its own reward-hacking dynamics is the core engineering bet.

METR's 2025 industry report on reward-hacking in frontier models (Von Arx, Chan, Barnes) is qualitatively consistent with Skalse 2022 and Pan 2022, though informal.

There is also a distinct threat the audit literature has begun to quantify: **eval-awareness** — the model's recognition that it is being evaluated rather than serving a real user, relevant because models sometimes behave differently when they detect evaluation — where the model recognizes it is being audited and modulates its behavior. Anthropic's "Measuring and improving coding audit realism with deployment resources" (alignment.anthropic.com, 2026-03-23) introduces *realism win rate* — the rate at which auditors achieve a verdict that holds up under fuller human review; concretely, how often an LLM judge mistakes a synthetic audit transcript for a real deployment transcript — and reports that grounding the auditor in real deployment resources (system prompts, tool definitions, codebases) lifts realism win rate on reward-hacking scenarios from 4.6 percent to 32.8 percent — a roughly 7x relative improvement, though 32.8 percent still means most synthetic transcripts remain distinguishable from real ones. Anthropic's **Petri 2.0** note (alignment.anthropic.com, 2026-01-22) reports a complementary mitigation: a prompted realism classifier that flags implausible auditor actions plus manual seed revision yielded a 47.3 percent median relative drop in eval-awareness on Claude models. Eval-awareness is a different failure mode from contamination — both can inflate apparent scores, but the mechanism (model knows it is being tested vs. model has seen the test) is distinct. The capstone's choice of a real-internet single-prompt directive inherits some realism by accident; whether the realism-win-rate metric should be adopted explicitly is an open design question.

### 6.3 The closest published peers to the capstone integrity audit

The runtime-with-internet integrity audit is an underexplored area. The two closest direct precedents are **NIST's CAISI cheating-evaluation work** and **AISI's Inspect Scout transcript-analysis pipeline**. NIST CAISI (the US National Institute of Standards and Technology; CAISI is NIST's AI Safety Institute, which has begun publishing methodology for evaluating whether AI agents cheat on tests — 2024-2025 web series, "Cheating in AI Agent Evaluations" at nist.gov/caisi/cheating-ai-agent-evaluations) built an Inspect-based transcript review system — Inspect is the UK AISI's open-source evaluation framework (used by Anthropic for some alignment audits), designed precisely for the kind of structured tool-use-transcript walks (transcript: the full log of every command the agent ran, every file it read, every web page it fetched, in order) the capstone's audit needs to do. The system runs an LLM reviewer over an evaluation transcript and scores it for cheating signals. NIST reports finding both solution-contamination and grader-gaming in their internal evaluations.

The operational sibling on the UK side is **AISI Inspect Scout** (AISI + Meridian Labs, 2026-02-25 blog post "A pipeline for transcript analysis using Inspect Scout," PyPI release 2026-04-29) — an open-source tool, built on Inspect, that defines a seven-step pipeline — define purpose, organize transcripts, sample, refine the question, design scanners, validate, apply — and ships scanners for refusal, eval-awareness, and tool errors. It is designed to scale to thousands of transcripts in parallel, and is the published methodology nearest in shape to what the capstone's audit does at a smaller scale. The companion AISI case study, "Transcript analysis for AI agent evaluations" (aisi.gov.uk, 2025-10-10), applies a similar pipeline to 6,390 samples from 71 cybersecurity capture-the-flag tasks across nine models, surfacing three quantified failure modes (refusal at 10-30 percent of runs, task abandonment, tool-call non-compliance). The case study is the closest published data point on what transcript-walk auditing actually finds on an agent-on-CTF task shape; it does not directly measure the audit's sensitivity to runtime-internet-lookup, which is the threat the capstone targets.

Where NIST CAISI and Inspect Scout audit one transcript at a time, **Meerkat** (Stein, Brown, Hassani, Naik, Wong, "Detecting Safety Violations Across Many Agent Traces," arXiv:2604.11806, 2026-04-13) takes the cluster-audit axis those two precedents do not, combining clustering with agentic search to surface natural-language-specified violations across large trace collections. Meerkat reports finding nearly 4x more reward-hacking examples on CyBench than prior audits (per the arXiv preprint; not yet peer-reviewed) and discovering developer cheating on a top agent benchmark. NIST CAISI and Inspect Scout audit one transcript at a time; Meerkat audits many at once, looking for clusters. The plan's discovery supervisor accumulates traces across many runs, so Meerkat is the published peer for the across-runs cluster-audit shape — adjacent rather than identical to `pr-e2b7fdf`'s single-transcript walk.

NIST CAISI, Inspect Scout, and Meerkat are closest on both the threat model (catching agents that consult forbidden materials at runtime) and the audit architecture (walk a transcript, check against rubric, emit a structured artifact). The plan's `pm_core/bench/audit.py` (described in `pr-e2b7fdf`) is the same single-transcript shape; the seven-step Inspect Scout pipeline is a published methodology the capstone could explicitly inherit or contrast with.

The empirical anchor for the human-review-as-final-gate framing is **Anthropic's "Pre-deployment auditing can catch an overt saboteur"** (alignment.anthropic.com, 2026-01-28), which underpins the framing the plan already adopts. In a blind auditing game with three trained-saboteur models and two benign baselines, the automated auditing agent's quantitative metrics and report alone caught 1 of 3 saboteurs; manual transcript review caught all three. The authors are explicit that overt sabotage is the easy case — eval-aware sabotage and subtle sandbagging (the model deliberately underperforming so its true capabilities are not measured) remain open problems. This is the empirical published rationale for the plan's pattern of producing a structured audit artifact for human review rather than treating the LLM-monitor verdict as final.

On the supervisor-architecture axis, **Anthropic's 2025 "Building and evaluating alignment auditing agents"** (alignment.anthropic.com/2025/automated-auditing/, research note from Anthropic, the company that makes Claude) is adjacent but addresses a different problem: alignment auditing (checking whether an AI model behaves the way its designers intended, especially where it might be tempted to misbehave), not cheating detection during evaluations. Anthropic introduces three agents that autonomously audit other agents — an Investigator, an Evaluation Agent, and a Breadth-First Red-Teaming Agent (red-teaming is deliberately attacking a system to find weaknesses before a real adversary does) — to surface misalignment in target models. The investigator agent identifies the right root cause about one time in eight by itself (10 to 13 percent); aggregating multiple parallel investigators raises that to roughly two in five (42 percent in a super-agent aggregation). The multi-axis decomposition (investigator / evaluator / red-teamer) is suggestive for how the capstone's audit might evolve. As a research note rather than peer-reviewed publication, some specific claims may shift.

A different measurement angle — structurally adjacent rather than directly on threat-model — comes from **ImpossibleBench** (Zhong, Raghunathan, Carlini 2025, arXiv:2510.20270, "ImpossibleBench: Measuring LLMs' Propensity of Exploiting Test Cases"), which constructs "impossible" task variants where the natural-language spec contradicts the unit tests, so any pass implies the agent took a specification-violating shortcut (typically by modifying the test). On Conflicting-SWEbench (full scaffold), GPT-5 cheats on 54.0 percent of runs (Zhong et al. 2025, §4 results) — i.e., more than half of attempted tasks resolve via test-modification rather than legitimate code change when the spec and tests conflict. No analogous numbers exist for runtime-internet lookup detection. ImpossibleBench is cited as a structurally adjacent measurement effort, but its sensitivity figures are for test-modification (an artifact-visible cheat — the patch contains the modification), not for external lookup (a trace-only cheat — the patch is fine, the URL is in the transcript), and the proxy direction is unknown: lookup-detection could be either easier (URLs in the trace are distinctive) or harder (the artifact contains no fingerprint) than test-modification detection. The audit's sensitivity is an open question and a measurement obligation, not a known floor.

There is also adjacent work on deception detection (**Liars' Bench**, Schwettmann et al. 2025, arXiv:2511.16035 — evaluates lie detectors for language models) and on "implicit reward hacking" detection (arXiv:2510.01367, "Is It Thinking or Cheating?"). These ask whether the agent is lying in its outputs, where the plan asks whether the agent consulted forbidden inputs. Useful as background, not as direct precedent.

In short: NIST CAISI and AISI Inspect Scout are the two direct precedents on threat model and audit architecture. Meerkat is the direct precedent on the across-traces cluster-audit axis. The Anthropic overt-saboteur work is the empirical anchor for the audit-plus-human-review pattern. The 2025 Anthropic auditing-agents note and ImpossibleBench are adjacent on different problems — alignment-auditing investigators and test-modification detection, respectively. `pr-e2b7fdf` is an integration job: NIST's and AISI's transcript-walk shape, sharpened by the Anthropic note's parallel-investigator pattern, complemented by Meerkat's cluster-audit pattern at the across-runs grain, tempered by the open question of LLM-monitor sensitivity in this setting.

## 7. Watcher Architectures (industry-dominated)

The plan's three watchers look a lot like Dependabot. The closest peers here are industry products; the academic adjacencies are real but thinner. This section maps what the plan borrows and where it diverges.

The industry baselines are **Dependabot** (GitHub, 2017-) and **Renovate** (Mend, 2017-): both continuously watch a repository's dependency surface, open PRs for updates, and integrate with the project's existing CI gates. They differ in scope (Dependabot supports 30+ package ecosystems, Renovate supports 90+) and configurability (Renovate has richer policy control), but architecturally they share the shape the plan reuses: a polling background process that scans a known surface, files PRs as it finds work, and leaves the merge decision to whatever gate — human or automated — already exists. Auto-merge bots like Mergify and Kodiak occupy the merge-gate slot. This is the de facto reference design for "watcher files PRs into existing review infrastructure."

Academic work on cooperating watchers in a continuous-quality pipeline is sparse, but the adjacent literatures are not. **AgentSpec** (Wang, Poskitt, et al. 2025, arXiv:2503.18666, ICSE 2026) is a customizable runtime-enforcement DSL for LLM agents. *DSL* — domain-specific language — is a small specialized language for expressing one kind of thing precisely (SQL is a DSL for database queries, CSS for styling web pages). AgentSpec's DSL expresses rules an agent must obey at runtime: rules are written as (trigger, predicate, enforcement) triples, and the paper reports preventing unsafe execution in over 90 percent of code-agent cases. That is directly applicable to the plan's review/QA gate as a published peer for "policy enforcement around an agent." **AgentAuditor** (Wang et al. 2025, arXiv:2506.00641, NeurIPS 2025) is a memory-augmented LLM-evaluator framework for safety and security review of agent traces — the published analog to the watcher review session (`pr-e84b43c`), because both involve a separate process reading captured traces to score them. Continuous fuzzing infrastructure (Google's **OSS-Fuzz** / ClusterFuzz, Serebryany USENIX Security 2017) is the closest analog from outside the LLM literature: continuous, durable corpus, files findings against existing review surfaces.

The plan's distinctive choice is *dynamic prioritization without persisted structured priority*. Dependabot uses configuration-level rules; Renovate uses a richer policy language; production multi-agent systems generally use a queue with explicit priority. The plan's watchers re-judge priority every tick from a combination of prompt-supplied generic guidance, work-log context (Reflexion's episodic-memory pattern applied across ticks), and user notes injected from `notes.txt`. There is no *structured* priority field; priority signals persist informally in the work-log and are re-weighted each tick. That is exactly the kind of state a structured priority field would normally hold, and the plan's bet is that prose persistence plus LLM re-weighting beats structured-field maintenance.

The case for the bet: priority is better re-derived from current context than maintained as state. Imagine the project just suffered a production outage. A stored priority list written yesterday wouldn't know. A priority that re-derives each tick reads the latest notes and reshuffles. "Current context" includes the latest regression-run result, the latest user note, and the latest review verdict — all of which can move quickly relative to a slow-changing priority field, which tends to drift.

The case against it is that re-deriving priority every tick costs LLM calls and risks oscillation if the context signals conflict. The cap on watcher work-per-tick and the work-log's append-only history are the secondary defenses against that pathology. The literature offers no direct precedent for this commitment, which is the plan's most distinctive structural choice.

Plotting the plan against its industry and academic neighbors on two axes — *live internet access vs. no internet access*, and *snapshot benchmark vs. continuous-loop operation*:

```
              snapshot benchmark │ continuous loop
                                 │
live          SWE-Bench          │ Dependabot
internet      SWE-Bench Pro      │ Renovate
access        SWE-Bench MM       │ this plan (capstone PR)
                                 │
              ───────────────────┼───────────────────
                                 │
no internet   ProgramBench       │ OSS-Fuzz / ClusterFuzz
access                           │ SapFix / Getafix
                                 │
```

The plan sits in the upper-right quadrant — live internet and continuous — which is sparsely populated in both the academic literature and in industry. Dependabot and Renovate are the closest neighbors and operate in a narrower surface (dependency updates). The capstone is what makes the plan continuous-and-internet-enabled; without it the plan is just continuous-and-offline (which OSS-Fuzz already does).

## Conclusion

The plan turns a regression-test suite into the inner clock of a continuous quality loop, and that simple framing hides where the research bets are. The autonomous regression and bug-fix loop draws on several research areas at once: benchmarks (§1), agents (§2), test generation (§3), program repair (§4), self-improving loops (§5), integrity auditing (§6), and watcher architectures (§7) each contribute well-understood pieces. The plan combines these pieces in a way that is mostly conventional at the part level and meaningfully novel at the integration level.

Where the plan most clearly reuses prior work: the agent itself is Claude Code, used as a black box behind a curated prompt surface in the spirit of SWE-agent's ACI argument. The scenario-quality supervisor is a verifier-guided generation pattern (Cobbe 2021, Lightman 2023) shaped to defend against Pan 2024's reward-hacking critique; its current `--resume`-based context arrangement is the most under-validated mitigation in the plan, and `pr-98f670e`'s design is the place a future cycle most needs to strengthen (see `pm/docs/adversarial-review/RESUME_DESIGN_NOTE.md` for a sketch). The three-watcher shape is a polling-bot pattern in the Dependabot/Renovate tradition with Reflexion-style episodic memory in `pm/watchers/*.log`. The bug-fix flow is automated program repair in the GenProg / Sapfix lineage with the generation mechanism swapped from genetic programming to LLM, and structurally similar to a Self-Debug loop (Chen et al. 2023), but at the watcher-tick level rather than the prompt-iteration level. The fakes infrastructure is standard test-double design applied to the LLM-and-GitHub boundary.

Where the plan departs from prior work: regression tests are **durable** (§3 — the drafted regression survives the run and becomes corpus, in contrast to throwaway test suites from CodaMosa or Fuzz4All); priority is re-derived from context every tick rather than persisted as structured state (§7); and, most consequentially, the capstone PR's realistic single-prompt evaluation inverts ProgramBench's offline assumption and substitutes a post-run integrity audit (`pr-e2b7fdf`, §6) — though as §1 notes, ProgramBench is itself a young benchmark whose own evidence base is still thinner than SWE-Bench's, so the contrast is between methodological bets, not between two equally-validated benchmarks.

The capstone's contribution is no longer "a novel post-run audit architecture" — that architecture was established in 2024-2025 by NIST CAISI and AISI Inspect Scout, both transcript-review pipelines for catching cheating modes in agent evaluations. The contribution is *applying that architecture to a different threat*: runtime-internet lookup-of-the-answer during autonomous agent runs, distinct from the alignment-auditing setups (AuditBench) and transcript-review-for-cybersecurity-cheating setups (NIST CAISI, AISI Inspect Scout) the precedents address. The residual novelty is threefold: the threat-model match (post-run audit applied to runtime-internet-lookup specifically, not to hidden-behavior auditing or CTF-task cheating); the allowlist mechanism specific to internet-permission contexts (curated `materials/` plus a written list of permitted external sources, against which the trace is checked); and the integration of the audit into the autonomous loop's verdict pathway (the audit verdict feeds the merge gate, not a standalone report). If the plan succeeds, the capstone is the most ambitious integration of AISI/NIST-line transcript-audit machinery with a continuous-quality loop; if it fails, it remains the most likely point of failure. AuditBench's tool-to-agent gap is a published reason to scrutinize the scenario-quality supervisor's verdict-aggregation step (`pr-98f670e`); Meerkat covers the across-traces cluster-audit axis the single-transcript precedents do not; Anthropic's 2026 overt-saboteur note is the empirical anchor for the audit-plus-human-review pattern the plan already adopts; ImpossibleBench is a structurally adjacent measurement effort whose numbers do not transfer cleanly.

Whether "durable" earns the stronger framing of "compounding" depends on measurements the plan does not yet collect. Operational definitions for "compounding" would need to address: (a) scenario count growth *weighted by the supervisor's quality verdict* to avoid spam (depends on `pr-98f670e`); (b) coverage delta per scenario with explicit accounting of diminishing returns; (c) bug-recurrence rate with denominator (recurrences per known fix merged, per week). These are aspirational metrics the plan owes — not currently collected — and the lit review treats "durable" as the defensible claim today, "compounding" as the future claim contingent on collecting these.

## Appendix: Terms used in this review

Plain-English glosses for the terms the review leans on most.

- **LLM (large language model)** — the kind of AI system, trained on text, that can read instructions and produce text in response. ChatGPT is the most famous example.
- **Claude** — the LLM made by the company Anthropic. The plan uses Claude as the engine behind every automated action.
- **Claude Code** — a separate product from Anthropic. It is a command-line tool, built on top of Claude, that lets the model edit files and run commands on a real computer. The plan's automated processes invoke Claude Code under the hood. Where the review names a specific model version ("Claude Opus 4.7"), it means a specific release of Claude.
- **pull request (PR)** — the standard mechanism in collaborative coding for proposing a change. One person (or process) writes "here is the change I made and why," and a reviewer accepts or rejects it before the change is folded into the shared codebase.
- **merge** — accepting a pull request, folding the proposed change into the shared codebase.
- **CI (continuous integration)** — the system that automatically runs a project's tests every time someone proposes a change.
- **QA (quality assurance)** — the work of checking that a change actually does what it claims and doesn't break anything else.
- **UX (user experience)** — the quality of how the software feels to use, separate from whether it is technically correct.
- **regression-test suite** — the durable checks that catch old bugs from returning, exercised every time a change is proposed.
- **watcher** — in the plan, a background process that wakes up periodically, looks at the state of the project, and files pull requests for work it thinks should happen. The plan defines three watchers.
- **the loop** — the repeated cycle a watcher runs: wake up, look at recent activity, decide whether to file a PR, sleep, wake up again.
- **implementation watcher** — a watcher that takes an existing pull request and tries to make the code changes that PR requires. The plan has two: one for bug fixes, one for UX improvements.
- **discovery supervisor** — a watcher that runs regression tests on a schedule and files new pull requests for whatever bugs or coverage gaps those tests reveal.
- **QA review loop** — the part of the plan that grades a candidate change: it runs the regression tests, watches the change in action, and decides whether to accept or reject.
- **capstone** — the final pull request in the plan (`pr-e2b7fdf`). It is an evaluation, not a feature: an automated agent has to complete a realistic engineering task under a written directive, with real internet access, and an audit afterward decides whether the agent actually did the work or cheated by looking the answer up.
- **auto-sequence** — a chained sequence of watcher actions, triggered by one keypress or one command, that runs through implementation, review, and QA without further human intervention.
