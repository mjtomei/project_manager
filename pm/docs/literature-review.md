# Literature Review: Autonomous Regression and Bug-Fix Loop

## 1. Introduction

This review is organized around a plan — `pm/plans/plan-regression.md` — to build a software-improvement loop staffed by three human-level roles: a **designer** that decides what to improve, a **software engineer** that implements it, and a **QA engineer** that verifies it, running together with no human driving the iteration. The design → build → QA cycle runs without a person naming the flow, reviewing each change, or deciding what to fix next. A human is not eliminated outright — software built for humans keeps a final taste-and-acceptance sign-off — but leaves the inner loop. The target is deliberately modest: the straightforward polishing and maintenance work a competent engineer could do given enough time, not frontier engineering.

Whether that is new, and what existing work it can draw on, is what the rest of the review takes up. Today's systems each fill part of the loop — the autonomous coding loops of Section 4e staff the software-engineer role well, but a human still supplies the design and the planning, and their QA is thin — and no surveyed system staffs all three roles together. The plan is a bet at the whole loop; it does not succeed yet, and pm has not closed it. The standing question against every piece of related work here is *which of the three roles does it staff, and how much of the loop runs without a human?*

The plan is implemented in the `pm` project-manager tool — `pm` is a command-line tool that tracks a project's planned work as a list of "pull requests" (the standard mechanism in collaborative coding for proposing a change: one person or process writes "here is the change I made and why," and a reviewer accepts or rejects it before the change is folded into the shared codebase). Terms like LLM, watcher, PR, and capstone are glossed in the Appendix on first use.

**What the plan does, in plain English.** The plan installs three background processes ("watchers"). A discovery supervisor schedules regression-test runs on a cadence — a regression-test suite being the standing set of automated checks a project runs to confirm that yesterday's fix did not break today's feature — and the regression runs themselves file any new bugs and coverage gaps they surface. The other two watchers each drain a pool of that filed work: one fixes bugs, one improves the user experience, each driving its items through a reproduce-fix-verify cycle with a separate review step that grades the candidate change. Put another way: coding agents already automate the developer; **the plan tries to automate a good human QA engineer and layer that role on top of the already-automated developer.**

Two evaluators sit inside the loop. A "scenario quality supervisor" checks that the review step itself was thorough enough before a change is allowed to merge. A capstone evaluation is a single-prompt engineering task with real internet access, followed by an audit that walks the agent's transcript to verify the agent did the work rather than searching for the answer online.

Watchers and the work loop at a glance:

```
a regression run is triggered   (on a timer, or inside a
        │                        QA loop running for another fix)
        │  files bug / improvement PRs via `pm pr add`
        ▼
two PR pools          bug pool (plan=bugs)   improvement pool (plan=ux)
        │
        │  discovery supervisor — a loop that reconciles
        │  the pools post-hoc (dedup, routing)
        ▼
two implementation watchers, each its own loop:
        bug-fix watcher          drains the bug pool
        improvement-fix watcher  drains the improvement pool

  each tick, a watcher picks the best PR from its pool and drives it:

        reproduce → fix → verify
              ▼
        QA review loop            runs the regression, grades the change
              ▼
        scenario quality supervisor   was the QA itself thorough enough?
              ▼
        merge        bug-fix: auto-merge on PASS
                     improvement-fix: held for a human taste check
              │
              └──▶ watcher loops back, picks the next PR next tick
```

Because the QA review loop runs regression tests, it can itself be the "QA loop running for another fix" at the top: a fix under QA can surface a fresh bug, which gets filed and drained like any other — the loop feeds itself.

**How this review is organized.** It is organized as a set of comparisons, not as a topic survey. Section 2 argues why the *harness* — the scaffolding and tooling around the model — is the right unit to measure progress in. Section 3 compares pm's harness against the other coding-agent harnesses, stage by stage, in a single table. Section 4 takes pm's distinctive components one at a time and compares each against the specialist tools that target the same job. Section 5 covers what to measure against — benchmarks and the contamination problem. Section 6 is an honest accounting of what is reused, what is adapted, and what is genuinely new. Section references cite the relevant PR ids from the plan throughout.

## 2. The harness is the contribution

The plan cannot train a frontier model — that takes resources only a few labs have — so its contribution has to be the *harness*: the scaffold of tooling, prompts, file access, and step-by-step structure wrapped around an existing model. The LLM (large language model — the kind of AI system, trained on text, that reads instructions and produces text in response) is the engine; the harness is the chassis. Recent evidence makes that a defensible place to bet. Across 2026 analyses of agent scaffolding, swapping the harness around a fixed model swings coding-benchmark scores by tens of points, while swapping the frontier model under a fixed harness moves the score by roughly one point — in one reported case a cheaper model on a better scaffold outscored a flagship model on its own vendor's scaffold (see References, "Agent scaffolding"). If the harness is where the points are, the harness is where a project without a training budget should work.

**ProgramBench is the benchmark this plan measures against, because of what it holds fixed.** ProgramBench (the 200-task benchmark where Claude Opus 4.7 with the minimal mini-swe-agent scaffold currently tops the leaderboard — programbench.com snapshot 2026-05-14; arXiv:2605.03546) gives the agent only a compiled executable plus user-facing documentation and asks it to *rebuild* the codebase from scratch, offline, inside a sandbox (an isolated environment that reaches the rest of the computer or the internet only when explicitly allowed), with no decompiler. Crucially, ProgramBench *fixes the harness*: every model runs inside the deliberately minimal mini-swe-agent scaffold, so the score reflects the model's own architectural reasoning rather than the surrounding tooling. That fixed harness is exactly the variable this plan moves. The plan's contribution is a *general harness* — the autonomous regression loop, the leader pass that synthesizes the task envelope (`pr-b3b8df0`), the compounding QA flow — and the claim it must clear is that this harness lifts the same frontier models above the standard mini-swe-agent leaderboard score. So the contribution is neither a new model nor a benchmark built from scratch, but a richer harness, measured against the standard-harness leaderboard, on a task chosen so a harness — not a bigger model — can move it.

A caveat on the measuring stick itself: ProgramBench is much newer than the SWE-Bench family, cited here from its website and a single arXiv preprint, and lacks the follow-up literature SWE-Bench has accumulated. Claims about its design should be weighted accordingly. Section 5 places it in the wider benchmark landscape.

## 3. Harness vs harness

This is the central comparison. Treat each coding-agent harness as a pipeline of internal stages and ask, stage by stage, what each harness does. The table below lays out seven established harnesses plus pm against eleven stages. "Localize" means finding where in the codebase to work; "Plan" means deciding the approach before editing; the rest are self-explanatory or glossed below.

| Internal stage | mini-swe-agent | SWE-agent | Agentless | Confucius | OpenHands | Claude Code | pm harness |
|---|---|---|---|---|---|---|---|
| Localize | model | ACI search | hierarchical | yes | model | model | per-PR scope |
| Plan | no | no | fixed pipeline | planning ext | no | plan mode | yes (plan files + leader pass) |
| Edit | bash | validated+linter | search/replace | validated | typed tools | Edit tool | yes |
| Run existing tests | model may | model may | yes | model may | model may | model may | yes (regressions) |
| Generate new tests | no | no | reproduction only (single bug) | test-gen (single issue) | no | no | yes (durable new-regression authoring) |
| Behavioral review of the change | no | linter (syntax) | patch ranking | no | safety only | no | yes (review step grades each change) |
| QA: discover & exercise user flows | no | no | no | no | no | no | yes (discovery supervisor + auditor) |
| Grade test thoroughness (test-the-test) | no | no | no | no | no | no | yes (scenario quality supervisor) |
| Fix the product when QA fails | no | no | no | no | no | no | yes (bug-fix / improvement watchers) |
| Integrity audit (cheat detection) | no | no | no | no | no | no | yes (capstone audit) |
| Continuous operation | no | no | no | no | no | session | yes (watchers on a cadence) |

The harnesses in the table, briefly. **mini-swe-agent** is the deliberately minimal scaffold ProgramBench uses as its baseline (mini-swe-agent.com). **SWE-agent** (Yang et al. 2024, NeurIPS 2024) is partly an agent and partly a paper arguing that the **agent-computer interface** — ACI, the menu of commands the agent has, such as file-open, scoped-edit, search, run-tests — matters as much as the model. **Agentless** (Xia, Deng, Dunn, Zhang 2024, arXiv:2407.01489) removes the agent loop entirely: a fixed three-phase pipeline of localize, repair, validate, which beat many agent systems on SWE-Bench Lite at $0.70 per task. **Confucius** is the custom-scaffold coding agent from the 2026 scaffolding analyses, notable for a planning extension and per-issue test generation. **OpenHands** (Wang et al. 2024, ICLR 2025; SDK paper arXiv:2511.03690) is the open-source platform that is the de facto SWE-Bench Verified baseline, reporting a 72 percent resolve rate with Claude Sonnet 4.5; it supports hierarchical agents that delegate to other agents. **Claude Code** is Anthropic's practitioner command-line tool, built on Claude, that lets the model edit files and run commands; it is also the substrate every pm watcher runs on, not a peer pm competes with.

**The headline.** Every harness in the table shares the top four rows — localize, plan, edit, run existing tests. That is the developer half of the job, and it is well-served by the existing literature. pm's harness is differentiated **entirely in the bottom seven rows** — the QA-engineer half: generating durable new tests, behaviorally reviewing the change, discovering and exercising user flows, grading whether the tests were thorough, fixing the product when QA fails, auditing for cheating, and running continuously. That is a genuine unfilled gap. No harness in the table fills the bottom three-to-five rows at all.

**Where pm is *not* differentiated, stated plainly.** On the top four rows pm reuses standard designs and is arguably *less* mature than the specialists. Its localization is just per-PR scope, not the hierarchical search of Agentless or the ACI search of SWE-agent. Its edit step is ordinary tool-use, not the validated-and-linted edit of SWE-agent or Confucius. Its planning leans on plan files and a leader pass rather than a tuned pipeline. The plan inherits these from Claude Code and contributes nothing to them.

**Narrowing the QA-half claim honestly.** The nearest analogs in the bottom rows must not be overstated. Agentless and Confucius do generate tests — but only a *reproduction test* that confirms the single reported bug, not tests that discover or exercise user flows; that is why the table marks them "reproduction only (single bug)" rather than a full "yes." "Regression test selection," a phrase that appears in some harness descriptions, is only a *filter* on the repo's already-passing tests — it picks which existing tests to run, it does not author new ones. And the 2026 scaffolding literature shows where the high-scoring harnesses actually invest their effort: memory, search, and context management — *not* a verification or QA loop. So the gap pm targets is real and not merely unnoticed; the field has been optimizing the developer half. What pm has not yet shown is that filling the QA half works in practice — that is Section 6's open question.

## 4. Component deep-dives

This section takes pm's distinctive components — the bottom rows of the Section 3 table — one at a time, and compares each against the specialist tools built for that exact job. The pattern for each subsection is the same: what the prior art does, what pm does, and the honest residual.

### 4a. QA worker: flow discovery and durable regression authoring

pm's QA worker discovers a user flow, drives it, and — when no existing regression test covers it — authors a new one that persists (`pr-2680fbf`, `pr-c2397e2`). The specialists targeting this step fall into three groups.

**Browser-automation agents with built-in AI.** Playwright — the dominant browser-automation framework — shipped Test Agents in v1.56 (October 2025): a Planner that proposes a test plan, a Generator that writes the test code, and a Healer that repairs tests broken by UI changes (Playwright Test Agents, playwright.dev/docs/test-agents). What the prior art does: turns a *named* scenario into runnable test code and keeps it green. What pm does: discovers the scenario itself. The honest residual: Playwright's Planner needs a human to name the scenario *and* supply a seed test to work from — it does not decide what to test. And the Healer explicitly only patches the *test* when the app changes; it never patches the *product*. pm's regressions need no Healer-style selector-healing at all, because an LLM re-drives the flow live each run rather than replaying brittle recorded selectors.

**AI-native testing platforms.** Octomind and QA Wolf are commercial platforms that *do* autonomously discover flows. What the prior art does: discovers flows by *crawling the app surface* — coverage-driven, walking links and forms to find untested paths. What pm does: discovers flows that are *work-driven* — the PR in flight triggers the new test, so the test is authored against the change actually being made, not against the whole app. This is the precise narrowing of pm's flow-discovery claim: not "first to autonomously discover flows" (false — these platforms do), but "work-driven rather than crawl-driven discovery." The honest residual beyond that: no AI-native platform closes the loop to *fixing the product* when a flow fails, and none grades whether the test it wrote was thorough.

**LLM-guided fuzzing.** Fuzzing means automatically generating large numbers of weird or random inputs to see what breaks. The LLM-guided line includes FLARE (arXiv:2604.05289), ChatAFL (NDSS 2024), and in-browser fuzzing (arXiv:2510.13543). What the prior art does: uses an LLM to generate inputs that drive coverage into untested code. What pm does: drives *user flows*, not raw input streams, and persists the result as a named regression. The honest residual: fuzzing finds crashes and coverage gaps; it does not produce a durable, human-readable regression for a specific user-facing flow, and it does not fix anything.

**Unit-test generation (folded in from the older test-generation literature).** A decade of work generates *unit* tests — tests of one function in isolation. **CodaMosa** (Lemieux et al. 2023, ICSE 2023) pairs evolutionary test search with Codex, using the LLM as the escape valve when the classical search hits a coverage plateau — a pattern `pr-c2397e2` reuses at a coarser grain. **ChatTester** (Yuan et al. 2024, FSE 2024) is an early systematic study of ChatGPT as a test generator; **TestPilot** (Schäfer et al. 2024, IEEE TSE) is a clean prompt-skeleton baseline for JavaScript; **ChatUniTest** (Xie et al. 2023) adds a Generation-Validation-Repair loop discussed in 4c. The SWT-Bench reactive line — generating a test that reproduces a *reported* issue — is well-populated: **Issue2Test** (Mündler et al. 2025), **SWE-Tester** (arXiv:2601.13713, the training-side counterpart), **Otter / e-Otter++** (arXiv:2502.05368, arXiv:2508.06365), and **R2E** (Jain et al. 2024, ICML 2024, repository-level test generation with executable feedback) all operate in that reactive shape, where a defect report already exists. **TestExplora** (Microsoft Research 2026, arXiv:2602.10471) moves closest to pm because it tests the *proactive* case. It benchmarks proactive test generation — discovering latent defects with all defect signals hidden — and reports state-of-the-art LLMs reach only a 16.06 percent fail-to-pass rate. That rate is the share of tasks where the model wrote a test that fails on the buggy code and passes on the fixed code, about one in six attempts. What pm does that TestExplora does not: `pr-2680fbf` (a) persists the drafted regression to the corpus rather than scoring once, (b) runs inside a continuous watcher loop with coverage feedback rather than a one-shot benchmark, and (c) consults the discovery supervisor's running work log rather than a fixed documentation oracle. TestExplora's 16.06 percent ceiling is the strongest published prior-art number against which `pr-2680fbf`'s yield should be measured. The honest residual: none of this literature closes the loop to fixing the product, and the durability claim — that drafted regressions *compound* rather than merely accumulate — is unproven (Section 6).

The closest analog to durable test accretion is not a paper but **OSS-Fuzz** (Serebryany, USENIX Security 2017), Google's continuous fuzzing service, where a corpus grows over time and survives the run that generated it. OSS-Fuzz shows "durable" earns its keep only when measured — bugs found per week, corpus growth. The plan inherits the design pattern; whether it inherits the measurement discipline is Section 6's question.

A note on test doubles. The plan fakes the LLM and the version-control backend during its own tests (`pr-fbda1a8`, `pr-abcf70f` `FakeClaudeSession`, `pr-9603d04` `FakeGitHubBackend`). A test double is a stand-in for a real component so the test runs without the real thing ("mock" = strict stand-in that fails if called wrongly; "fake" = lightweight working substitute). A search for academic peers surfaced nothing peer-reviewed; this is industry-tooling territory (LiteLLM mock mode, the VCR.py record-replay pattern). pm's contribution here is only applying the classical mocks-vs-fakes distinction to the LLM-and-version-control boundary, absent academic guidance.

### 4b. Bug-fix watcher vs automated program repair

pm's bug-fix watcher (`pr-30588a7`, merged) detects a failing test, proposes a fix, verifies the test passes, and gates on review. That is **automated program repair**, a literature over a decade old.

What the prior art does. **GenProg** (Le Goues et al. 2012, IEEE TSE) evolves program variants by genetic programming — an older AI technique that breeds candidate variants like genes — until the test suite passes. **Sapfix/Sapienz** (Marginean et al. 2019, ICSE-SEIP) is Facebook's production deployment of the same pattern at million-line scale, with engineer-approval gates. **RepairAgent** (Bouzenia, Devanbu, Pradel 2025, ICSE 2025, arXiv:2403.17134) is the first repair work built as an autonomous LLM agent: it interleaves bug-information gathering, repair-ingredient gathering, and validation under a finite-state machine, reporting 164 Defects4J bugs repaired. **Getafix** (Bader et al. 2019, OOPSLA) extracts repair templates from past human fixes; **AutoCodeRover** (Zhang et al. 2024, ISSTA) uses the program's structural tree to localize the bug before handing it to the LLM. **Self-Debug** (Chen et al. 2023, arXiv:2304.05128) and **Self-Edit** (Zhang et al. 2023, ACL) have the model execute a candidate, observe the failure, and retry with that feedback.

What pm does. `pr-30588a7` reruns this playbook with the LLM as generation engine, closest to RepairAgent's localize → analyze → generate → verify → iterate loop. But it runs at watcher-tick cadence (minutes apart) rather than per-bug, with a persistent work log between attempts. It is also integrated into the discovery → implementation → review chain, so fixed bugs feed back to the discovery supervisor's queue.

The honest residual. pm's bug-fix step is *not* a contribution over RepairAgent on the repair itself — it lacks RepairAgent's finite-state orchestration and its Defects4J validation. What is new is only the *placement*: the repair step sits inside a continuous loop that also discovers the bug and audits the fix. The repair, in isolation, is reused prior art.

### 4c. Scenario quality supervisor (test-the-test) vs verifier-guided generation

pm's scenario quality supervisor (`pr-98f670e`) grades whether the QA review step was thorough enough before a change may merge — it tests the test.

What the prior art does. **Verifier-guided generation** trains or prompts a separate model whose only job is to grade candidates. **Cobbe et al. 2021** (arXiv:2110.14168, the GSM8K paper) shows a separately-trained verifier ranking candidate solutions beats a finetuned generator alone. **Lightman et al. 2023** ("Let's Verify Step by Step," NeurIPS 2023) extends this to *process supervision* — scoring each reasoning step, not just the final answer. ChatUniTest's Generation-Validation-Repair loop (Xie et al. 2023) is the test-generation instance: it catches and fixes compile errors and assertion failures before returning a test.

What pm does. The supervisor is a separately-prompted Claude session whose only job is to grade the QA session's evidence depth — a verifier in the Cobbe/Lightman tradition, applied to test thoroughness rather than to math answers. It is deliberately *not* the same model critiquing its own work with no new information ("intrinsic self-correction"), the setting **Self-Refine** (Madaan et al. 2023, NeurIPS 2023) explored and **Huang et al. 2023** ("Large Language Models Cannot Self-Correct Reasoning Yet," ICLR 2024) showed can *degrade* performance. Instead, each pm iteration produces fresh external artifacts (regression files, transcripts, coverage reports) that mechanical checks verify — the regime where self-correction reliably helps. **Reflexion** (Shinn et al. 2023) is the closest agent-side ancestor — its written "lessons learned" memory is the pattern the per-watcher work-logs reuse across ticks. The multi-agent frameworks **AutoGen** (Wu et al. 2023) and **MetaGPT** (Hong et al. 2024) are the nearest structural analogs to pm's separated writer-and-checker roles, though pm's watchers are more tightly scoped and tick rather than dialogue.

The honest residual and the caution. The risk is **reward hacking** — a model scoring well on a measurement without doing the underlying task, for example modifying the test instead of fixing the bug. **Pan et al. 2024** ("Spontaneous Reward Hacking in Iterative Self-Refinement," arXiv:2407.04549) shows this emerges spontaneously in self-refinement loops, *more readily when generator and evaluator share context*. Skalse et al. 2022 and Pan, Bhatia, Steinhardt 2022 are the peer-reviewed theoretical backing. pm's supervisor reduces shared context by running as a separate session that reads the QA session's captures and tool-use transcript via Claude Code's `--resume` mechanism. But `--resume` still carries forward the full session thread (system prompt, prior turns, tool-use history), so the separation is partial. The amendment cap (default 2) is the primary mitigation; two amendments is exactly the regime Pan et al. found measurable hacking in, so the cap is necessary but not sufficient. This makes the supervisor the most under-validated mitigation in the plan; a stronger redesign (fresh session, captures-only input, optionally an ensemble verdict) is sketched in `pm/docs/adversarial-review/RESUME_DESIGN_NOTE.md`. Anthropic's **AuditBench** (alignment.anthropic.com, 2026-03-10) documents an adjacent failure: a *tool-to-agent gap* where a checker reading good evidence still fails at the aggregation step. The threat differs (hidden-behavior auditing, not QA grading), but the structural concern — verdict aggregation can fail even on good evidence — applies to `pr-98f670e`.

### 4d. Integrity audit vs alignment-auditing pipelines

pm's capstone runs with real internet access, so the agent could simply look up the answer. The capstone audit (`pr-e2b7fdf`) walks the agent's tool-use transcript afterward — the full ordered log of every command run, file read, and page fetched — and checks every URL against an allowlist (a written list of which external sources the agent may consult).

What the prior art does. **NIST CAISI** (the US National Institute of Standards and Technology's AI Safety Institute; "Cheating in AI Agent Evaluations" web series, nist.gov/caisi) and **AISI Inspect Scout** (AISI + Meridian Labs, 2026; an open-source pipeline built on the Inspect evaluation framework) both run an LLM reviewer over an evaluation transcript to score it for cheating signals. **Meerkat** (Stein et al., arXiv:2604.11806, 2026) takes the across-runs axis — clustering plus agentic search to surface violations across large trace collections, reporting nearly 4x more reward-hacking examples on CyBench than prior audits. **AuditBench** documents the verdict-aggregation failure mode discussed in 4c. On the model-side disclosure axis, Anthropic's **Introspection Adapters** (2026, arXiv:2604.16812) train an add-on that makes a model self-report its learned behaviors — complementary to the capstone's run-time transcript audit rather than a substitute for it. Adjacent deception-detection work (**Liars' Bench**, Schwettmann et al. 2025, arXiv:2511.16035) asks whether an agent is lying in its outputs, where the capstone asks whether it consulted forbidden inputs — useful background, not direct precedent.

What pm does. `pm_core/bench/audit.py` (in `pr-e2b7fdf`) is the same single-transcript walk-and-check design, with the audit verdict feeding the merge gate of an autonomous loop rather than producing a standalone report.

The honest residual. The audit *architecture* is not novel — NIST CAISI and Inspect Scout established it in 2024-2025. Three things are residual. (1) **Threat-model match**: the precedents target hidden-behavior alignment auditing (AuditBench) or known cheating modes on cybersecurity capture-the-flag agents (NIST CAISI, Inspect Scout); the capstone applies the same architecture to runtime-internet lookup of a reference solution — a different threat with different signals (URLs in the trace vs. behavioral patterns in outputs). (2) **Allowlist-against-trace boundary**: the curated `materials/` directory plus a written list of permitted sources is the boundary every URL gets checked against; the precedents do not specify this because their threats are not internet-fetching. (3) **Integration into the verdict pathway**: no precedent sits inside an autonomous loop's merge pathway this way. The honest size of this contribution: an integration job on an established architecture, not a step into an empty field. The runtime-enforcement and trace-evaluation literature is adjacent — **AgentSpec** (Wang et al. 2025, ICSE 2026) expresses runtime rules for LLM agents and **AgentAuditor** (Wang et al. 2025, NeurIPS 2025) is a memory-augmented evaluator of agent traces — but neither targets the internet-lookup threat. The audit's detection rate on this specific threat is unmeasured. The nearest floors come from three benchmarks. **ImpossibleBench** (Zhong et al. 2025, arXiv:2510.20270) measures how readily models exploit test cases (GPT-5 cheats on 54 percent of conflicting-spec runs), but for an artifact-visible cheat, not a trace-only one. **RHB** (Thaman 2026) reports base shortcut-exploit rates of 0–13.9 percent across 13 frontier models. And **TRACE** (Deshpande et al. 2026, arXiv:2601.20103) reports 45–63 percent detection for state-of-the-art LLM judges catching code-environment reward hacks. If the capstone's recall on runtime-internet-lookup falls below 50 percent in pilot, the audit reverts from a merge gate to a human-review trigger — the falsifiable claim `pr-e2b7fdf` must clear.

### 4e. Continuous operation: the autonomous-loop landscape

pm's three watchers run continuously and re-judge what to work on every tick. The peers fall into three groups, and the useful comparison is not whether they run autonomously — several do — but which of the three roles (§1) each loop staffs: a designer that decides what to improve, a software engineer that implements it, a QA engineer that verifies it.

**Mechanical-loop tools.** **Dependabot** (GitHub, 2017-) and **Renovate** (Mend, 2017-) poll a repository's dependency surface, open PRs for updates, and defer the merge to whatever gate exists. **OSS-Fuzz** (§4a) continuously finds crashes. **SWE-CI** (Chen et al. 2026, arXiv:2603.03823) benchmarks maintainability across an average of 71 consecutive commits per repository, but offline against historical traces — "continuous" there is continuous integration over commit history, not a live loop. These run with no human in the loop, but only on mechanical work: a dependency bump is a version string; a fuzzer finds a crash and fixes nothing. None of the three roles is really staffed — there is nothing to design and nothing to verify beyond the existing test gate.

**Human-triggered coding agents.** The background coding agents — GitHub's Copilot coding agent (generally available September 2025), Cursor's background agents, GitHub Agent HQ (February 2026), and Devin (Cognition) — staff the software-engineer role on open-ended work: take a task, edit across files, run tests, open a pull request. But a human files or assigns the issue at the front (the designer role) and reviews the draft PR at the back (standing in for QA). The engineer is automated; the other two roles are the human.

**Autonomous execution loops.** Closest of all is the loop pattern that emerged through 2025-2026: the "Ralph loop" (a continuous single-task loop with verification as its exit gate), "closed-loop development," and Cursor's "self-driving codebases." Reported runs are striking — an OpenAI Codex team shipped roughly a million lines across 1,500 pull requests with no human-written code; Cursor ran hundreds of agents building a browser in a week. These run continuously with no human driving each step, so the loop itself is plainly the field's natural next step. But they still staff one role well — the software engineer. The design and planning are supplied by a human up front (a spec, a task list). And the verification is thin — running the existing tests as the exit gate, or, in "closed-loop development," an explicit human UAT at the end. The loop is not the gap. The gap is which roles it staffs.

| Role | Mechanical-loop tools | Human-triggered agents | Autonomous execution loops | pm |
|---|---|---|---|---|
| Designer — decide what to improve | — (nothing to design) | human files the issue | human supplies the spec | discovery supervisor + improvement proposals, over the whole-project model |
| Software engineer — implement it | — (version bump) | automated | automated | bug-fix / improvement-fix watchers |
| QA engineer — verify it | existing test gate | human reviews the PR | thin (run tests) / human UAT | QA review loop + scenario quality supervisor |

**What pm does.** pm's watchers run continuously like the others, but the loop is built to staff all three roles at once. The watcher shape itself is Dependabot's; the open-ended fix is the background agents'; the continuous, no-human-driving loop is the Ralph pattern's — pm reuses each. Its bet is the *combination* — a designer, an engineer, and a QA engineer in one loop — plus two implementation choices the others do not make. The first is planning over the whole-project model: every watcher and QA session carries the full plan / PR / dependency graph via pm CLI, so the designer role works project-wide rather than task-locally. The second is dynamic prioritization with no persisted priority field, where priority signals live in the work-log as prose the next tick re-weights.

**The honest residual.** No single role's component is a contribution: §4a-§4d showed each has prior art, and the loop is the convergent next step. The residual is the *integration* — all three roles staffed together, on the polishing-and-maintenance class of work — and, above all, whether it runs as a working system, first on pm itself and then on other projects. That bar is Section 6's subject.

## 5. Benchmarks, contamination, and the capstone

This section covers what the plan measures itself against, and the contamination problem that threatens any such measurement.

**The benchmark lineage.** Coding benchmarks have three eras. The first — **HumanEval** (Chen et al. 2021, the Codex report) and **MBPP** (Austin et al. 2021) — graded isolated function-completion against hidden tests (small automated checks the candidate cannot see, so cannot game). They were rapidly contaminated: by GPT-4 both had appeared in enough scraped training corpora that scores were hard to interpret. **EvalPlus** (Liu et al. 2023, NeurIPS 2023) made the test-insufficiency problem explicit — extending HumanEval's tests ~80x dropped pass rates by 19–29 percent, meaning a fifth to a third of "passing" code was actually wrong. The second era responded to the realism gap: **SWE-Bench** (Jimenez et al. 2024, ICLR 2024) collects 2,294 tasks from real GitHub repositories, each asking for a patch (a diff — a line-by-line listing of what changed) that resolves an issue and passes a hidden test. **SWE-Bench Verified** is a 500-task human-filtered subset; **SWE-Bench Pro** (Deng et al. 2025, arXiv:2509.16941) is the contamination-resistant successor — 1,865 problems, long-horizon enterprise-style tasks, on which the strongest systems resolve well under half and, under cost constraints, roughly a quarter. **SWE-Bench Multimodal** (Yang et al. 2024) extends the shape to 617 visual JavaScript tasks.

The third era responds to a problem the second never solved: contamination. SWE-Bench tasks are real GitHub issues, so the resolutions are also on GitHub and likely in training data. In February 2026 OpenAI audited the 138 hardest tasks in SWE-Bench Verified and found at least 59.4 percent materially flawed, with frontier models reproducing gold patches verbatim from the task ID alone ("Why SWE-bench Verified no longer measures frontier coding capabilities," openai.com, 2026-02-23); OpenAI recommended SWE-Bench Pro as the replacement. **LiveCodeBench** (Jain et al. 2024) defends by tying problems to post-cutoff release dates; **BigCodeBench** (Zhuo et al. 2024) by library-rich calls that resist memorization; the "Saving SWE-Bench" mutation work (Garg et al. 2026, arXiv:2510.08996) by converting formal issue text into chat-style queries. **ProgramBench** (Section 2) belongs here — the rebuild-from-binary task with runtime internet denial.

**The contamination problem has two forms**, and the plan confronts both. *Pretraining contamination* asks: has the benchmark answer already appeared in the model's training data? — fixed at training time, detected after the fact. The literature on it (Sainz et al. 2023, Findings of EMNLP; Riddell et al. 2024, ACL, using edit-distance and structural comparison to catch renamed near-duplicates) now expects every benchmark to publish a contamination measurement, with time-windowed evaluation the leading defense. *Runtime contamination*, the form the capstone confronts, asks: *during the evaluation itself*, did the agent fetch the reference solution off the open internet? — decided turn by turn. These are different problems with different defenses. ProgramBench's defense is denial: no internet in the sandbox. The capstone's is the opposite bet — *grant* internet access (real engineers use the internet, so banning it measures an unrealistic situation) and *detect misuse afterward* with the post-run audit of Section 4d. A related threat the audit literature has begun to quantify is **eval-awareness** — the model recognizing it is being tested and modulating its behavior; Anthropic's coding-audit-realism and Petri 2.0 notes (alignment.anthropic.com, 2026) report mitigations. Eval-awareness differs from contamination — both inflate scores, but one is the model knowing it is tested, the other the model having seen the test.

**The capstone as the harness-vs-harness bet, cashed out.** The capstone (`pr-e2b7fdf`) is where Section 2's bet is settled: pm's general harness driving a frontier model, measured against the standard mini-swe-agent leaderboard, on an internet-enabled task variant. The audit is the mechanism that makes that variant scorable, not the headline. One genuine but young distinction: the capstone runs QA *inside* the harness that produces the submission — the harness checks its own work before declaring done — whereas SWE-Bench and ProgramBench grade externally and post-hoc, after the harness has finished. That is a real structural difference, but it is unproven and should not be oversold: an internal QA loop is only as good as the QA loop, and Section 4c flags that loop as the plan's most under-validated piece. An offline-shape offshoot (`pr-0cf3626`) keeps the ProgramBench offline constraint for leaderboard comparability.

## 6. Reused, adapted, and new

An honest contribution accounting, against the Section 3 table.

**Reused, and not differentiated.** The top-of-harness stages — localize, plan, edit, run existing tests — are reused from prior art and pm is arguably *less* mature there than the specialists. The agent is Claude Code, used as a black box behind a curated prompt surface in the spirit of SWE-agent's ACI argument. The bug-fix repair step (4b) is automated program repair in the GenProg/Sapfix/RepairAgent lineage with the LLM swapped in for genetic programming — reused, not new. The watcher-polling shape (4e) is Dependabot's. The test-double infrastructure is standard mocks-vs-fakes design. None of these is a contribution; the plan inherits them and says so.

**Adapted.** Several patterns are lifted and re-grained. Reflexion's verbal-reinforcement memory (Shinn et al. 2023) becomes the per-watcher work-log carried across ticks instead of agent trials. CodaMosa's "LLM as escape valve on a coverage plateau" becomes coverage-driven scenario growth. The Cobbe/Lightman verifier pattern becomes the scenario quality supervisor. The NIST CAISI / Inspect Scout transcript-walk becomes the capstone audit. Each is adapted, not invented — and Section 4 states for each what the adaptation does and does not add.

**New — the residual contribution, and the bar it has to clear.** The contribution is a *working* system, in two concrete senses. First, the **three roles staffed together as one loop**: a designer that decides what to improve (the discovery supervisor and the improvement-proposal path, working from the whole-project model pm holds), a software engineer that implements it (the bug-fix and improvement-fix watchers), and a QA engineer that verifies it (the QA review loop and the scenario quality supervisor) — no human driving the design → build → QA iteration. Section 4 showed each role's component has prior art, and §4e the continuous-loop pattern its convergent peers; no surveyed system staffs all three roles at once. Second — the real bar — the system has to **run on pm's own development first, then bootstrap onto other projects**: maintaining its own tooling, then generalizing via the leader pass that synthesizes a task envelope for an arbitrary project (`pr-b3b8df0`) to maintain projects it was not built for.

One adjacent 2026 line runs in the opposite direction — agents optimizing other agents' harnesses rather than a harness doing software work (VeRO, arXiv:2602.22480; "Agentic Harness Engineering," arXiv:2604.25850; "Meta-Harness," arXiv:2603.28052); a different problem, but the nearest neighbour to pm's self-hosting goal of a harness that maintains its own tooling.

The contribution is **unproven**. The plan does not yet collect the measurements that would defend it. "Durable" — that drafted regressions persist — is defensible today. "Compounding" — that they make later runs find more — is not; it depends on a measurement now committed: bugs-found-per-week-of-runtime across four consecutive watcher-week samples on a live project, with a week-over-week increase of at least 5 percent as the threshold. The discovery supervisor logs bug-find events, so it is the source. Until those four samples exist, "compounding" is a future claim. And the loop itself — that the three roles run well enough together to take the human out of the iteration — rests on the scenario quality supervisor (§4c), the plan's most under-validated component. The honest closing position is not a victory lap: the field is converging on the autonomous loop, the plan's bet is staffing it with all three roles and proving it first on pm itself and then on projects it was not built for, and whether the bet pays out is exactly what remains to be measured.

## Appendix: Terms used in this review

Plain-English glosses for the terms the review leans on most.

- **LLM (large language model)** — the kind of AI system, trained on text, that can read instructions and produce text in response. ChatGPT is the most famous example.
- **Claude** — the LLM made by the company Anthropic. The plan uses Claude as the engine behind every automated action.
- **Claude Code** — a separate product from Anthropic. It is a command-line tool, built on top of Claude, that lets the model edit files and run commands on a real computer. The plan's automated processes invoke Claude Code under the hood. Where the review names a specific model version ("Claude Opus 4.7"), it means a specific release of Claude.
- **harness** — the scaffold of tooling, prompts, file access, and step-by-step structure wrapped around an LLM. The LLM is the engine; the harness is the chassis. The plan's contribution is a harness.
- **pull request (PR)** — the standard mechanism in collaborative coding for proposing a change. One person (or process) writes "here is the change I made and why," and a reviewer accepts or rejects it before the change is folded into the shared codebase.
- **merge** — accepting a pull request, folding the proposed change into the shared codebase.
- **CI (continuous integration)** — the system that automatically runs a project's tests every time someone proposes a change.
- **QA (quality assurance)** — the work of checking that a change actually does what it claims and doesn't break anything else.
- **UX (user experience)** — the quality of how the software feels to use, separate from whether it is technically correct.
- **regression-test suite** — the durable checks that catch old bugs from returning, exercised every time a change is proposed.
- **watcher** — in the plan, a background process that wakes up periodically, looks at the state of the project, and files pull requests for work it thinks should happen. The plan defines three watchers.
- **the loop** — the repeated cycle a watcher runs: wake up, look at recent activity, decide whether to file a PR, sleep, wake up again. "Closing the loop" means removing the human from it entirely.
- **implementation watcher** — a watcher that takes an existing pull request and tries to make the code changes that PR requires. The plan has two: one for bug fixes, one for UX improvements.
- **discovery supervisor** — a watcher that runs regression tests on a schedule and files new pull requests for whatever bugs or coverage gaps those tests reveal.
- **QA review loop** — the part of the plan that grades a candidate change: it runs the regression tests, watches the change in action, and decides whether to accept or reject.
- **capstone** — the final pull request in the plan (`pr-e2b7fdf`). It is an evaluation, not a feature: an automated agent has to complete a realistic engineering task under a written directive, with real internet access, and an audit afterward decides whether the agent actually did the work or cheated by looking the answer up.
- **auto-sequence** — a sequence of automated steps — write the code, review it, run the tests — triggered by a single keystroke and running without further input. The plan uses this to chain implementation, review, and QA on a new pull request.
